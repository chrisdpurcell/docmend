## Findings

### ISSUE-001: Mutation and recovery journals do not cover every commit boundary

- first_pass: `1`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - `src/docmend/writer/apply.py:483-503` writes a durable intent only for `rename_and_rewrite`; ordinary rewrite/rename operations mutate at `:503-517` and append their only applied record at `:562-583`.
  - `src/docmend/restore.py:232-284` performs every inverse mutation before appending its inverse record, and multi-step inverse operations have no intent/reconciliation protocol.
  - A kill after restore writes the original but before it removes/replaces the target leaves a superset that a rerun rejects at `src/docmend/restore.py:178-185` as an original-path collision.
  - `docs/specs/docmend.md:687,704-706` promises progress-to-date records and says a crash cannot orphan a completed mutation.
- impact: a kill, process abort, disk-full event, or manifest append/fsync failure can leave applied or restored state without the recovery pointer needed for mechanical restore, audit, or deterministic resume. Interrupted restore can remain safe from byte loss yet fail to converge automatically.
- recommendation: use a prepare/intent plus terminal record for every apply and restore mutation class. Include before/after identity, backup references, overwrite state, root identity, and a deterministic reconciliation rule for each interruption boundary.
- required_tests: kill or injected append failure immediately after pure rewrite, pure rename, each restore step, and before every terminal record; prove deterministic adopt/complete/refuse behavior.

### ISSUE-002: Restore and verify trust manifest paths outside the recorded source root

- first_pass: `1`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - `src/docmend/cli.py:379-394` chooses the restore lock root from the first record only.
  - `src/docmend/restore.py:48-80,83-103,146-258` reads and mutates absolute manifest paths without checking each record against a single resolved `source_root`.
  - `src/docmend/verify.py:88-114` hashes every manifest target without binding it to the requested verify PATH or a consistent recorded root.
  - The manifest schema permits arbitrary non-empty path strings and nullable `source_root` values.
- impact: a wrong, concatenated, corrupted, or crafted manifest can restore outside the intended library; verify can report clean while checking a different tree.
- recommendation: treat manifests as untrusted input. Require one consistent canonical root for modern ledgers, enforce every original/target path beneath it at read time and immediately before mutation, bind verify PATH to it, and require an explicit trusted root/migration path for legacy manifests.
- required_tests: mixed roots, outside-root original/target, wrong-tree verify, parent symlink swap, legacy explicit-root handling, and restore refusal before any mutation.

### ISSUE-003: Hash and containment preconditions are not held through mutation

- first_pass: `1`
- severity: `high`
- confidence: `medium`
- issue_type: `implementation`
- evidence:
  - Apply reads/hashes and resolves a pathname at `src/docmend/writer/apply.py:363-380`, then later stats and mutates through fresh pathname operations at `:503-517`.
  - Restore hashes at `src/docmend/restore.py:83-103`, then later stats and mutates through the path at `:153-258`.
  - Existing run locks coordinate docmend processes, not an editor, sync client, or other local writer.
- impact: another process can replace a file or parent path after validation, allowing newer bytes or a different object to be overwritten despite a passing stale-hash/containment check.
- recommendation: bind validation, backup, transform, and commit to the same opened object identity using descriptor-relative/no-follow operations where supported, or revalidate type, device/inode, containment, and digest at the last safe commit boundary.
- required_tests: deterministic object replacement for rewrite, rename, rename-and-rewrite, and restore; parent-symlink swap at commit; hard-link identity policy.

### ISSUE-004: A late overwrite collision bypasses the preservation gate

- first_pass: `2`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - `src/docmend/writer/gate.py:116-135` requires preservation only for overwrite targets that exist during gate evaluation.
  - `src/docmend/writer/apply.py:389-396` rechecks target existence later, but if a target appeared after the gate it sets `clobber=True` without re-evaluating preservation.
  - `src/docmend/writer/apply.py:419-465` records target bytes but writes a backup only when `backup_root` is configured; `rename_overwrite` then destroys the late target.
  - Isolated reproduction: rename-only overwrite plan, no strategy, clean gate; create target after gate; apply reported `applied` and replaced the late file.
- impact: a concurrently created user file can be irreversibly destroyed even though the FR-005/FR-011 safety gate was intended to prevent unpreserved overwrite loss.
- recommendation: make overwrite preservation an action-time invariant. If any target exists at commit, require and verify an active preservation strategy before clobbering; without one, refuse/skip regardless of the earlier gate result.
- required_tests: target creation after gate for pure rename, rename-and-rewrite, tool-backup, external-preservation, and no-strategy modes.

### ISSUE-005: Verify ignores unreadable and timed-out scan candidates

- first_pass: `3`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - Discovery records `unreadable` and `timeout` skips at `src/docmend/discovery.py:313-331,435-446`.
  - `src/docmend/cli.py:843-865` creates findings from `inventory.files` plus optional artifacts, never from `inventory.skipped`.
  - The resulting summary can say `0 files checked, 0 findings` and exit `0` when every candidate was skipped.
- impact: unreadable or pathological documents disappear from the verification population while automation receives a clean verdict.
- recommendation: convert error-class scan skips into verify findings and report candidate, checked, policy-excluded, unreadable, and timeout counts separately. Never return clean when requested candidates could not be checked.
- required_tests: one unreadable file, one timeout, mixed checked/skipped set, all candidates skipped, and policy-only exclusions.

### ISSUE-006: Apply/report/verify accounting can hide unattempted plan actions

- first_pass: `2`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - Collision policy `fail` breaks the action loop at `src/docmend/writer/apply.py:634-677`; later actions have no outcome.
  - The report model has no terminal run state or `not_attempted/aborted` outcome (`src/docmend/report.py:19-86`).

### ISSUE-007: Verify does not validate recorded recovery backups

- first_pass: `3`
- severity: `high`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - `src/docmend/verify.py:88-114` checks only applied target `after_sha256` values.
  - It never reads `backup_path`, `before_sha256`, `overwritten_backup_path`, or `overwritten_sha256` even though those fields are the recovery evidence.

### ISSUE-008: Whole-run structures violate the binding bounded-memory contract

- first_pass: `4`
- severity: `high`
- confidence: `high`
- issue_type: `convention-misalignment`
- evidence:
  - SPEC NFR-001 at `docs/specs/docmend.md:355` requires memory independent of corpus size and no whole-corpus in-memory structures.
  - `src/docmend/discovery.py:222-235,425-475` retains all inventory records; `src/docmend/planning.py:189-229,392-413` retains pending/actions/skips/path sets; `src/docmend/writer/apply.py:615-704` retains every outcome.
  - `src/docmend/artifacts.py:105-188` materializes whole document/model representations when reading and writing JSON artifacts.
  - `tests/test_scale.py:39-51,194-205` explicitly accepts `64 MiB + 10 KiB/file`, approximately 1 GiB at 100k, which is the opposite asymptotic contract.
  - The opt-in scale test is skipped by the normal gate and currently fails at 1,000 files because its manifest-count assertion predates intent records.
- impact: file-count growth can exhaust memory beyond the one measured corpus while the traceability matrix still marks NFR-001 complete.
- recommendation: make an owner-level contract decision through spec/ADR change control: either adopt a measured O(file-count) metadata budget and supported maximum, or stream/spool artifacts and outcomes to satisfy the current invariant. Test multiple corpus counts and actual CLI stage boundaries.
- required_tests: several file counts, peak-memory slope, artifact serialization/deserialization, full verify, and a repaired intent-aware scale lifecycle assertion.

### ISSUE-009: Terminal report publication is outside preflight, lock scope, and full durability

- first_pass: `3`
- severity: `medium`
- confidence: `high`
- issue_type: `implementation`
- evidence:
  - The safety gate preflights the manifest directory only (`src/docmend/writer/gate.py:184-232`), not a custom report destination.
  - Apply releases the run lock at `src/docmend/cli.py:537-582`, then writes the report at `:584` without a guarded error path.
  - `src/docmend/artifacts.py:93-103` fsyncs the temporary file but does not fsync the containing directory after replacement.
- impact: apply can mutate and journal the corpus, then fail to publish its terminal report after releasing coordination; the user may receive an unhandled exception before artifact locations/terminal status are printed. Power loss can also lose a newly created report directory entry.
- recommendation: preflight the selected report path, keep the lock through terminal publication, catch/report late failures with manifest location and mutation status, and fsync the report directory after replace where supported.
- required_tests: unwritable/full report destination after successful mutation, injected report replace/fsync failure, and terminal messaging/lock lifetime assertions.

### ISSUE-012: Accepted configuration values are operationally inert

### ISSUE-019: Destructive engines can be called without the safety coordinator

## Open Questions Or Assumptions

- Is the supported threat model a quiescent single-user tree, or must docmend tolerate editors, sync clients, and attacker-modifiable directories during mutation? ISSUE-003 remains at least a correctness race even under benign concurrency.
- Which local/removable/network/case-insensitive/Unicode-normalizing filesystems are supported, and what crash/power-loss durability is promised on each?
- Must verify certify recoverability as well as live output correctness? The approved zero-loss/recovery posture and recorded backup fields imply yes; ISSUE-007 assumes that interpretation.
- Should NFR-001 remain constant-working-memory/no-whole-corpus, or should the owner approve an explicit O(file-count) metadata budget and maximum supported count?
