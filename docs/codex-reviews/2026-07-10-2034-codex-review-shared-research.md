## Shared Stack And Ecosystem Guidance

### Python packaging, dependencies, and tests

- Treat `pyproject.toml`, `uv.lock`, build artifacts, console entry points, and the supported Python range as one release contract. In CI, `uv sync --locked`, `uv run --locked`, or `uv lock --check` verifies lock freshness; `--frozen` merely trusts the existing lock and can miss metadata drift. Include every dependency group the gate claims to cover, and pin the uv executable version as well as its setup Action. uv also documents that the lock is tool-managed and should be committed, not hand-edited ([uv locking and syncing](https://docs.astral.sh/uv/concepts/projects/sync/), [uv in GitHub Actions](https://docs.astral.sh/uv/guides/integration/github/)).
- Test what users install, not only an editable checkout. Build both an sdist and wheel, ensure the wheel can be built from the sdist, inspect their contents, install in a clean environment, run the CLI smoke/contract tests, and verify packaged schemas and other data files. Where local/workspace source overrides exist, `uv build --no-sources` checks public publishability. PyPA describes the sdist/wheel flow and recommends publishing both ([Packaging Flow](https://packaging.python.org/en/latest/flow/), [Package Formats](https://packaging.python.org/en/latest/discussions/package-formats/), [`build` basic usage](https://build.pypa.io/en/latest/how-to/basic-usage.html), [uv package builds](https://docs.astral.sh/uv/guides/package/)).
- Keep unit, boundary, end-to-end, crash/restart, and scale evidence distinct. Use historical artifact fixtures for compatibility, property/state-machine tests for multi-step resume/restore behavior, and deterministic synthetic corpora for encoding/path edge cases. Do not use retries to conceal flaky tests; pytest identifies uncontrolled system state and order dependence as common causes ([pytest good practices](https://docs.pytest.org/en/stable/explanation/goodpractices.html), [pytest flaky tests](https://docs.pytest.org/en/stable/explanation/flaky.html), [Hypothesis stateful testing](https://hypothesis.readthedocs.io/en/latest/stateful.html)).
- A clean vulnerability scan is necessary but incomplete. `pip-audit` finds known package advisories; it explicitly is not a static analyzer, does not detect malicious packages, and may not see vulnerabilities in bundled system libraries. Review provenance, maintainership, transitive dependencies, licenses, platform wheels, and compensating controls separately ([pip-audit security model](https://github.com/pypa/pip-audit#security-model)).
- Current uv also offers lock-aware `uv audit --locked`. The repository convention assigns the gate to `pip-audit`, so a child may compare coverage but should not silently substitute tools without changing that contract ([uv audit](https://docs.astral.sh/uv/reference/cli/#uv-audit)).
- Keep public dependency bounds separate from the exact development/CI resolution, and review custom indexes/direct URLs for dependency-confusion, credential, and mutable-source risk. uv's default `first-index` strategy is a deliberate defense; `unsafe-best-match` changes that posture ([uv package indexes](https://docs.astral.sh/uv/concepts/indexes/)).

### CLI, artifact, and schema contracts

- Inventory all externally observable contracts together: commands and options, stdout/stderr separation, help text, exit codes, config, structured logs, plans/manifests/reports, JSON Schemas, backup layout, and resume/restore semantics. Compatibility tests should cover both valid historical producers and malformed/adversarial consumers.
- Give every durable artifact an explicit schema identity/version and define its compatibility window. Decide whether unknown fields are preserved, ignored, or rejected; distinguish missing from null; bound arrays/strings and integer ranges; and reject ambiguous duplicate JSON member names. RFC 8259 notes that duplicate-name handling is unpredictable across implementations ([RFC 8259](https://www.rfc-editor.org/info/rfc8259/)).
- Pin the JSON Schema dialect. In Draft 2020-12, `format` is annotation-only by default unless the assertion vocabulary or validator configuration enables validation; `additionalProperties` and `unevaluatedProperties` also have different composition behavior ([JSON Schema 2020-12](https://json-schema.org/draft/2020-12), [validation vocabulary](https://json-schema.org/draft/2020-12/json-schema-validation)).
- A “data migration” review should not assume a database. File artifacts, frontmatter, manifests, and installed package data are schemas too. Require explicit upgrade/rejection behavior, golden compatibility fixtures, and a recovery path before breaking changes ship.

### Filesystem safety, atomicity, and recovery

- Treat user-supplied roots, config/artifact paths, discovered entries, symlinks, junctions, mount points, hard links, and destination collisions as trust boundaries. Lexical prefix checks are not containment checks, and a canonical path checked earlier can change before use. Review containment and file identity at the actual mutation point ([CWE-22 path traversal](https://cwe.mitre.org/data/definitions/22.html), [CWE-59 link following](https://cwe.mitre.org/data/definitions/59.html), [CWE-367 TOCTOU](https://cwe.mitre.org/data/definitions/367.html)).
- Python explicitly warns that `os.path.commonprefix()` can return an invalid path and that `os.access()` followed by `open()` creates a security hole. `commonpath()` is component-aware but still does not freeze a mutable tree. For hostile/shared trees, use descriptor-relative, per-component no-follow operations or an OS containment primitive, then inspect the opened object; Linux `O_NOFOLLOW` alone protects only the final component ([Python `os.path`](https://docs.python.org/3/library/os.path.html), [Python `os`](https://docs.python.org/3/library/os.html), [Linux `openat2(2)`](https://man7.org/linux/man-pages/man2/openat2.2.html)).
- Secure mutation normally means: atomically create a temporary file with restrictive permissions on the destination filesystem using `TemporaryFile`, `NamedTemporaryFile`, `mkstemp`, or equivalent—never `mktemp` or generate-name-then-open; write and validate all bytes; flush; apply intended metadata; when crash durability is promised, `fsync` after the final data/metadata change; atomically replace; then `fsync` the containing directory where supported. Handle `EXDEV`, disk-full, permission, and delayed writeback errors explicitly ([Python `tempfile`](https://docs.python.org/3/library/tempfile.html), [Linux `rename(2)`](https://man7.org/linux/man-pages/man2/rename.2.html), [Linux `fsync(2)`](https://www.man7.org/linux/man-pages/man2/fsync.2.html)).
- Atomic replacement gives visibility atomicity, not automatically power-loss durability, end-to-end backup integrity, or multi-file transactionality. Plans, backups, manifests, and originals need a documented commit order and an interruption result for every boundary between steps.
- At apply time, verify the plan precondition against the opened object's type, identity, and digest; abort on scan/apply drift. Backup and transform those exact opened bytes rather than reopening by pathname between checks.
- Define one policy for symlinks, hard links, and non-regular objects across discovery, plan, apply, backup, restore, and verify. Reject FIFOs, devices, and sockets by default unless the product explicitly supports them. No-follow controls do not contain hard links: where strict root containment is promised, use copy-and-replace semantics or reject multiply linked files. Treat a saved manifest as untrusted input during restore and reapply containment and identity checks at that boundary.
- Do not infer “no change” from a failed remote-filesystem operation. For example, the Linux `rename(2)` manual notes that NFS can report failure after the rename occurred. Recovery therefore needs identity/checkpoint evidence and retry-safe reconciliation, not only exception handling.
- Define filename identity independently from document-content normalization: case sensitivity, Unicode normalization, confusables, reserved names, undecodable POSIX names, and collision policy vary by filesystem. Preserve the original spelling or raw-byte round trip needed for manifests and test platform-specific aliases ([PEP 383](https://peps.python.org/pep-0383/)). Unicode UTS #39's confusable skeleton is an internal comparison form, not a filename normalization, storage, or display form ([Unicode UTS #39](https://www.unicode.org/reports/tr39/)).
- Do not normalize document content to simplify filename handling. Unicode UAX #15 warns that compatibility normalization can erase semantic distinctions; preserve actual filesystem names and use normalization/case folding only for an explicitly documented collision policy ([Unicode UAX #15](https://unicode.org/reports/tr15/)).

### Observability and performance for a batch CLI

- Use a stable event vocabulary and one correlation/run ID across scan, plan, apply, restore, and verify. Useful fields include event name, timestamp, severity, stage, outcome, error type, elapsed time, counts and bytes, schema version, and safe document identifiers. OpenTelemetry's stable log data model is a useful interoperability reference even without adopting its SDK ([OpenTelemetry Logs Data Model](https://opentelemetry.io/docs/specs/otel/logs/data-model/)).
- A batch workflow needs evidence of last success/failure, total and per-stage duration, work processed, error counts, retries/resumes, and terminal status. Monitoring must distinguish “not run” from “ran and succeeded” ([Prometheus instrumentation guidance](https://prometheus.io/docs/practices/instrumentation/)).
- Benchmark before optimizing. Use representative synthetic corpus shapes and record platform/filesystem, cold versus warm cache, wall time, throughput, per-stage timing, peak memory, and tail/outlier behavior. Preserve correctness checks in the benchmark and profile the measured bottleneck ([Python profilers](https://docs.python.org/3/library/profile.html)).

### CI, supply chain, and release

- Give `GITHUB_TOKEN` read-only defaults and only per-job permissions that are required. Pin every non-local Action and reusable workflow, including `actions/*`, to a full commit SHA. Do not let privileged `pull_request_target`/`workflow_run` jobs check out untrusted code or consume untrusted artifacts/caches; treat scripts, build tools, and test plugins as code execution. GitHub calls a full SHA the only immutable Action release reference ([GitHub Actions secure-use reference](https://docs.github.com/en/actions/reference/security/secure-use)).
- Gate dependency changes in pull requests as well as periodically scanning the resolved environment. GitHub's dependency review compares dependency deltas, but current dependency-graph documentation does not list `uv.lock`; verify actual graph ingestion/submission rather than assuming resolved-lock coverage. Native Dependabot support for `uv` updates is separate, and update automation should cover both `uv` and `github-actions` ecosystems ([GitHub dependency review](https://docs.github.com/en/code-security/concepts/supply-chain-security/dependency-review), [dependency-graph ecosystems](https://docs.github.com/en/code-security/reference/supply-chain-security/dependency-graph-supported-package-ecosystems), [Dependabot ecosystems](https://docs.github.com/en/code-security/reference/supply-chain-security/supported-ecosystems-and-repositories)).
- Prefer PyPI Trusted Publishing with a narrowly scoped release workflow, job-level OIDC permission, and a protected environment/manual approval. The workflow and everything it executes become part of the publishing credential boundary. Build once, test that exact artifact, then publish it rather than rebuilding in the publish job. `pypa/gh-action-pypi-publish` generates publish attestations by default; `uv publish` uploads adjacent attestations but does not generate them ([PyPI Trusted Publisher security model](https://docs.pypi.org/trusted-publishers/security-model/), [producing attestations](https://docs.pypi.org/attestations/producing-attestations/), [uv publishing](https://docs.astral.sh/uv/guides/package/)).
- Attestations bind an artifact to an identity/provenance statement; they do not prove that the code is safe. Verify the published artifacts against the expected repository/workflow and distinguish a PyPI publish attestation from stronger build provenance ([PyPI attestation security model](https://docs.pypi.org/attestations/security-model/), [SLSA Build Track](https://slsa.dev/spec/v1.2/build-track-basics)).
- A release gate should bind version, tag, source commit, lockfile, changelog, schema compatibility, built contents, install smoke test, vulnerability disposition, and attestation. Define yanking/rollback and data-recovery procedures before release; a package rollback cannot undo documents already mutated by an older executable.
- If GitHub Releases are part of delivery, current immutable releases lock the tag and assets and create a release attestation; assemble all assets as a draft before publishing ([GitHub immutable releases](https://docs.github.com/en/code-security/concepts/supply-chain-security/immutable-releases)).

### Sensitive-data and privacy baseline

- The sensitivity labels justify protective defaults; they do not by themselves establish HIPAA, GLBA, GDPR, or another legal regime. Map the data lifecycle first—source, memory, temp files, backups, manifests, logs, crash output, test artifacts, CI uploads, issue reports, and any telemetry—and determine jurisdiction, roles, retention, and deletion requirements with the owner. NIST's Privacy Framework is a voluntary risk-management tool, not a legal determination. Version 1.0 remains final while 1.1 is an initial public draft ([NIST Privacy Framework](https://www.nist.gov/privacy-framework)).
- Prefer local/offline processing. Remote conversion, telemetry, and crash upload should be explicit opt-in and content-free by default unless a separately reviewed data-flow decision justifies more.
- Minimize collection and persistence. File paths, names, titles, frontmatter, snippets, hashes of low-entropy values, and exception text may all identify a person or document. Prefer random/run-scoped IDs, aggregate counts, and allowlisted error fields. Redact document bytes/snippets, passwords, tokens, keys, connection strings, environment values, raw exception payloads, and subprocess output before every sink. Sanitize CR/LF, delimiters, and control characters before display; ensure debug/stack traces cannot bypass redaction; protect logs from unauthorized read/alteration; and test logging failure modes ([OWASP Logging Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html), [OWASP Secrets Management](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html)).
- Public fixtures and CI corpora should be independently created and conspicuously fictional. Public-domain sources address copyright, not privacy, and require explicit privacy/legal screening and minimization. Masking or deleting obvious identifiers does not prove safe de-identification; NIST recommends explicit goals, disclosure-risk assessment, and re-identification analysis ([NIST SP 800-188](https://csrc.nist.gov/pubs/sp/800/188/final)).

### Recovery and incident baseline

- Runbooks should cover interrupted mutation, wrong-root or containment breach, corrupt/mismatched manifest, restore failure, bad release/schema change, sensitive-data disclosure, and compromised dependency/release workflow. For each: detection signal, stop/contain condition, owner, evidence to preserve, recovery steps, verification, communication, and post-incident follow-up ([AWS runbook guidance](https://docs.aws.amazon.com/wellarchitected/latest/framework/ops_ready_to_support_use_runbooks.html), [NIST SP 800-61 Rev. 3](https://csrc.nist.gov/pubs/sp/800/61/r3/final)).
- Recovery evidence is stronger than backup creation evidence. Exercise restore on safe data and verify availability, completeness, integrity, and the selected recovery point. CISA's organization-level ransomware guidance recommends offline/encrypted backups and regular availability/integrity testing; that is a threat-model reference, not automatically a CLI rollback-copy requirement ([CISA StopRansomware Guide](https://www.cisa.gov/stopransomware/ransomware-guide)).
- Bind manifests to the selected root and prefer relative paths. Verify backup bytes before restore, detect conflicts with newer user work, and verify the restored bytes afterward. A digest detects change but does not authenticate a backup when an attacker can rewrite both data and manifest.
- Backups inherit the originals' sensitivity: use restrictive permissions, deliberate placement and retention, and avoid unintended cloud-synced temp/cache locations.
- Describe cleanup honestly: ordinary unlinking is logical deletion, not a portable secure-erasure guarantee on SSD, copy-on-write, removable, or network storage. Media sanitization belongs to the OS/device lifecycle ([NIST SP 800-88 Rev. 2](https://csrc.nist.gov/pubs/sp/800/88/r2/final)).
- Integrate preparation, detection, response, and recovery with ordinary risk management rather than treating the runbook as a release-time appendix ([NIST SP 800-61 Rev. 3](https://csrc.nist.gov/pubs/sp/800/61/r3/final)).

### Conditional AI baseline

Apply this only if a child confirms model inference, retrieval, prompt construction, or agent/tool execution in the product. Documents are untrusted prompt input: indirect prompt injection can be embedded in text or markup. Never send corpus content or metadata to a hosted model without an explicit data-flow decision; treat model output as untrusted; enforce authorization and path/operation policy outside the prompt; constrain tools and egress; require deterministic validation and human approval for destructive actions; and retain privacy-safe provenance and evaluation evidence ([NIST AI RMF Generative AI Profile](https://www.nist.gov/publications/artificial-intelligence-risk-management-framework-generative-artificial-intelligence), [OWASP LLM01 Prompt Injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/), [OWASP LLM02 Sensitive Information Disclosure](https://genai.owasp.org/llmrisk/llm022025-sensitive-information-disclosure/)).

### Conditional remote API, webhook, and job baseline

Apply this only if a child confirms a network/API/queue boundary. Give every remote call connect and total deadlines; retry only operations with known idempotent semantics, at one layer, with a bounded budget and exponential backoff plus jitter. An HTTP `202` is noncommittal and needs a status-monitor contract; machine-readable errors should be stable and privacy-safe ([RFC 9110 idempotent methods](https://www.rfc-editor.org/rfc/rfc9110.html#name-idempotent-methods), [RFC 9457 Problem Details](https://www.rfc-editor.org/rfc/rfc9457.html), [AWS timeout/retry guidance](https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/)).

For webhooks, verify signatures over the unmodified raw body, compare in constant time, use the provider-defined event/delivery identity for deduplication, allow out-of-order arrival, acknowledge before slow work, and monitor the queue and dead-letter path. Enforce freshness/replay windows only when the provider supplies a trustworthy timestamp or replay mechanism, and use the exact provider contract ([GitHub webhook practices](https://docs.github.com/en/webhooks/using-webhooks/best-practices-for-using-webhooks), [Stripe webhook behavior](https://docs.stripe.com/webhooks)). The IETF [`Idempotency-Key` draft](https://datatracker.ietf.org/doc/draft-ietf-httpapi-idempotency-key-header/) expired in April 2026 and must not be represented as a finalized standard.

- Define the scale/performance acceptance envelope: corpus distributions, storage media, cold/warm runs, time and memory budgets, and supported concurrency. Children may need targeted library/platform research after the bottleneck and deployment environment are known.
