# Combinatorial Testing Strategy for the Multi-Check Safety Gate

**Date:** 2026-07-05

**Related:** GAP-39 (external gap reference supplied by the calling task; no `GAP-` identifier scheme exists anywhere in this repo as of this research — see `docs/research/license-compliance-tooling.md` for the prior instance of this same note, and "Known drift found during this research" below) · OQ-005 (apply safety-gate checklist — this report's primary anchor) · OQ-014 (real-write CLI/config opt-in — one gate input) · `docs/specs/docmend.md` §7.1 FR-005/FR-006 (safety gate, backup/manifest), §7.2 NFR-004 (conservative-by-default), §8.5 (Design Constraints), §12.1 ERR-001–ERR-006, §17.2 Test Strategy, §17.3 FR-005 traceability row, §21 OQ-005 · `docs/open-questions.md` OQ-005's candidate checklist (the ~10-item set this report treats as the combinatorial "factors") · `docs/research/path-containment-toctou.md` (one gate check, output-path containment) · `docs/research/property-based-testing-hypothesis.md` (a related but distinct testing technique — see "Why not Hypothesis for this" below)

**Gap it fills:** OQ-005 already enumerates roughly ten independent conditions the apply safety gate must check before any write (plan validity, tool/schema version compatibility, source-hash freshness, explicit write opt-in, output-path containment, collision-policy satisfaction, risky-file skip/fail posture, preservation-strategy presence, manifest writability, backup-destination validity/containment), and §17.3's FR-005 row currently says only "Safety-gate refusal tests per missing strategy" — one test per check, evaluated alone. Nothing in the spec says how the checks compose: whether they are tested only one-at-a-time, how many of the `2^10`-plus joint states get covered, in what order the gate evaluates them, or what a caller can rely on when two or more checks fail at once and the report has to name _the_ blocking reason. Because docmend's entire safety case rests on this gate (§1: "safety mechanisms... are core requirements rather than conveniences"; NFR-004: "every destructive capability is opt-in"), an interaction bug here — two checks that silently depend on each other, or a nondeterministic pick of which failure gets reported — is exactly the class of defect combinatorial/interaction testing exists to catch, and exactly the class a >100k-file unattended batch run cannot afford to hit for the first time in production. This report closes that gap: it recommends a concrete pairwise/n-wise sampling technique and Python tool, a deterministic gate-evaluation and blocking-reason contract, and a §17.3 FR-005 test-strategy addition.

## Recommendation

1. **Adopt `allpairspy` as a dev-only test dependency** for combinatorial coverage of the safety gate's check matrix. It is pure Python, has zero runtime dependencies, integrates directly with `pytest.mark.parametrize` (docmend's existing test framework), and needs no JVM or native binary — unlike Microsoft PICT or NIST's ACTS, both of which are the more feature-complete reference tools in this space but require a non-Python toolchain docmend's dependency policy (§8.6, C-001: "Python 3.14+ with the repository's locked tooling stack") does not otherwise need. [community, github.com/thombashi/allpairspy]
2. **Test the gate in three layers, not one flat parametrize:** (a) one-factor-at-a-time exhaustive tests (today's §17.3 FR-005 row — keep it, it is cheap and still required), (b) a pairwise (`t=2`) combinatorial sweep across all gate checks generated by `allpairspy`, and (c) an escalation trigger to `t=3` scoped only to checks with a plausible coupling (see §4), following NIST's own documented "start at pairwise, raise the strength only if faults keep appearing" guidance rather than jumping straight to exhaustive or guessing at a higher strength up front. [official, NIST SP 800-142]
3. **Make the gate's ~10 checks pure, order-independent predicates evaluated in a fixed, documented sequence**, with the gate reporting _both_ the full set of failing checks (for the human/operator report, FR-018) _and_ one deterministic `blocking_reason` selected by a fixed priority list — not by "whichever check happened to run first" or "whichever raised an exception first." This is the same shape as AWS IAM's policy-evaluation contract (deny-precedence is a documented, order-independent rule, not an artifact of evaluation order) and avoids the class of bug pairwise testing is specifically good at catching: two checks whose outcomes silently depend on each other. [official, docs.aws.amazon.com IAM policy evaluation logic]
4. **Land this as a documented gate contract plus a §17.3 FR-005 traceability update**, not a new dependency-only decision — see §5 for the concrete spec-shaped addition and §6 for where it reconciles.

## 1. Why this is a combinatorial-testing problem, not a plain unit-testing problem

OQ-005's own "Agent notes" already lists the candidate safety-gate checklist docmend needs before any writing `apply` run:

1. Plan is valid against the current schema.
2. Plan tool/schema version is compatible.
3. Plan source hashes still match current files (FR-003).
4. Explicit write opt-in was provided (OQ-014, `--write`).
5. Output root/path containment passes (`docs/research/path-containment-toctou.md`).
6. Collision policy is explicit and satisfied (FR-011: `skip`/`fail`/`overwrite`).
7. Low-confidence/risky files are skipped, or the run is configured to fail before writing (FR-007/FR-015).
8. At least one byte-preserving preservation strategy is active (FR-005).
9. Manifest writing is enabled and writable (FR-006).
10. Backup destination, if used, is outside the mutation target path and writable (FR-006, OQ-005).

Treating each as a binary pass/fail factor gives `2^10 = 1,024` joint states before even accounting for multi-valued factors such as the three-way collision policy (`skip`/`fail`/`overwrite`) or the several distinct preservation-strategy backends (Git, external backup, tool-written backup) — which push the true state space well past 1,000. Exhaustively testing that state space is not proportionate for a v1 safety gate, but testing each check in isolation (today's spec language) only proves the checks work _alone_; it says nothing about whether check 6's containment logic is accidentally influenced by check 8's preservation-strategy state, or whether a specific two-failure combination produces a garbled or nondeterministic report. That gap between "each factor individually correct" and "the fault only appears when two or three factors combine" is the empirical justification for combinatorial (pairwise/n-wise) testing as a discipline, not just a Python library choice.

NIST's combinatorial-testing research group (Kuhn, Kacker, Lei) built the empirical case for this over a series of studies from 1999–2004 and later work: across multiple independently studied systems (a medical device, a web browser, a web server, network security software), **the large majority of reported faults were triggered by only one or two parameters acting together, and no fault in any of the studied systems required an interaction of more than six parameters** — Kuhn and Wallace's 2001 medical-device study found 98% of flaws were pairwise (2-way) interactions, with no fault requiring more than a 4-way interaction; a later web-server/browser study found no fault required more than a 6-way interaction. This is stated as an empirically-derived "interaction rule": most failures are induced by single-factor faults or by the joint combinatorial effect of a relatively small number of factors. [official, NIST SP 800-142 "Practical Combinatorial Testing"; NIST CSRC "Why do Combinatorial Testing?"]

The direct practical consequence NIST draws from this, and the one this report adopts for docmend, is a **staged strength policy**: start with pairwise (`t=2`) coverage, and only increase the interaction strength to `t=3` (or higher) for the specific factors where faults keep appearing, rather than either stopping at single-factor tests or jumping to exhaustive/high-strength coverage everywhere. NIST states this guidance directly: "start with pairwise testing, and continue increasing the interaction strength `t` until no errors are detected by the `t`-way tests, then (optionally) try `t+1`." [official, NIST SP 800-142]

NIST also documents the cost model that makes this tractable: the number of `t`-way combinatorial tests required is proportional to `v^t · log(n)` for `n` parameters with `v` values each — logarithmic in the number of factors, not exponential. For docmend's ~10 mostly-binary gate checks, this puts pairwise coverage on the order of a few dozen test cases at most (the exact count is generator-dependent — see §2), several orders of magnitude below the `1,024`+ exhaustive state space, while still exercising every pairwise interaction at least once. [official, NIST SP 800-142]

## 3. A deterministic gate-evaluation and blocking-reason contract

The research question's second half — "how should the gate deterministically report which failing check blocked?" — is a design-decision gap, not just a test-strategy gap, and it is what makes the checks combinatorially testable in the first place: **the checks must be pure, independent predicates over a snapshot of (plan, config, filesystem state) with no side effects and no dependency on each other's outcome.** If check 6 (path containment) or check 8 (preservation strategy) mutate shared state or short-circuit each other, pairwise testing cannot cleanly attribute an observed interaction failure to "these two factors together" — it becomes entangled with evaluation order instead.

Two real-world precedents for how mature systems resolve "many checks, one deterministic answer, need a legible reason":

- **AWS IAM's policy evaluation logic** documents a rule that is order-independent by design: an explicit deny anywhere in the applicable policy set overrides any allow, regardless of which policy the evaluator happens to consult first. The contract is stated as a property of the _outcome_ ("an explicit deny always overrides an allow"), not as "the deny statement evaluated first wins." [official, docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_evaluation-logic.html]
- **Kubernetes admission control** runs an ordered sequence of webhooks (mutating first, then validating), but documents that execution order is itself a controllable, named property (webhooks execute in lexical name order) rather than an accident of registration order, and that _any_ rejection from _any_ validating webhook blocks the request — the aggregate outcome is deterministic even though multiple independent checks run. [official, kubernetes.io/docs/reference/access-authn-authz/extensible-admission-controllers; community corroboration, r/kubernetes execution-order thread]

Applied to docmend's gate, the recommended contract is:

1. **Evaluate every check, every run — no short-circuiting.** All ~10 checks are cheap (hash comparisons, path resolution, config/flag reads — no network, no large I/O per §8.6/§13.1), so there is no performance reason to stop early, and evaluating all of them is what makes the full failure list available for the report (FR-018 requires "errors, skip reasons" to be recorded, not just the first one encountered).
2. **Each check is a pure function** `check(context) -> CheckResult(check_id, passed, reason)` over an immutable snapshot built once per apply invocation — no check may read filesystem/plan state that another check has not also been given, and none may mutate shared state. This is what makes checks independently unit-testable (today's §17.3 row) and pairwise-testable (§2) without evaluation-order noise.
3. **Define one fixed, documented priority list** for the ~10 check IDs (a plain ordered enum in code, not derived from dict iteration order or registration order). This list serves two purposes: it is the order checks are _listed_ in the report (stable, reviewable output), and it is the tiebreak rule for `blocking_reason` when more than one check fails simultaneously — `blocking_reason = first CheckResult with passed=False in priority order`. A defensible default priority (cheapest/most-foundational failure first, mirroring "no preservation strategy" as the single non-negotiable per §1/G-002): plan validity → version compatibility → hash freshness (FR-003) → explicit write opt-in (OQ-014) → path containment → collision policy → risk/confidence skip posture → preservation-strategy presence (FR-005) → manifest writability → backup-destination validity. This ordering is itself an implementation decision the owner should confirm alongside OQ-005, not something this report unilaterally settles.
4. **Report both**, not either/or: `blocking_reason` (singular, deterministic, for scripts/CI/quick diagnosis) and `all_failures: list[CheckResult]` (complete, for the human operator triaging a refused run without re-running with extra flags). JSON Schema validators exemplify the "report everything, not just the first" side of this: `jsonschema`'s `iter_errors()` (already the pattern used by `docs/research/managing-pandoc-markdown-and-strict-yaml-frontmatter.md` for frontmatter validation) returns every violation, not just the first, precisely because a user fixing one error at a time and re-running is worse UX than seeing the full list once. [official, json-schema.org validator behavior as already cited in the sibling frontmatter report]

A synthetic sketch (illustrative shape only — not a claim about final module structure, and using placeholder names, no real paths or library content per C-002):

```python
from dataclasses import dataclass
from enum import Enum, auto


class GateCheckId(Enum):
    """Fixed priority order — this Enum's declaration order IS the contract (see §3.3)."""
    PLAN_VALID = auto()
    VERSION_COMPATIBLE = auto()
    HASHES_FRESH = auto()
    WRITE_OPT_IN = auto()
    PATH_CONTAINMENT = auto()
    COLLISION_POLICY = auto()
    RISK_POSTURE = auto()
    PRESERVATION_STRATEGY = auto()
    MANIFEST_WRITABLE = auto()
    BACKUP_DESTINATION = auto()


@dataclass(frozen=True)
class CheckResult:
    check_id: GateCheckId
    passed: bool
    reason: str | None = None  # populated only when passed is False


@dataclass(frozen=True)
class GateOutcome:
    gate_passed: bool
    blocking_reason: CheckResult | None
    all_failures: tuple[CheckResult, ...]


def evaluate_gate(context) -> GateOutcome:
    checks = (
        _check_plan_valid,
        _check_version_compatible,
        _check_hashes_fresh,
        _check_write_opt_in,
        _check_path_containment,
        _check_collision_policy,
        _check_risk_posture,
        _check_preservation_strategy,
        _check_manifest_writable,
        _check_backup_destination,
    )  # order here MUST match GateCheckId declaration order (unit-tested, see §5)
    results = tuple(check(context) for check in checks)
    failures = tuple(r for r in results if not r.passed)
    return GateOutcome(
        gate_passed=not failures,
        blocking_reason=failures[0] if failures else None,
        all_failures=failures,
    )
```

The unit-testable invariant this enables — and that pairwise testing specifically exercises — is: **`blocking_reason` never depends on anything except which checks failed, never on the state of a check that passed.** That is exactly the kind of hidden-coupling bug a pairwise sweep across pass/fail combinations of the ten checks is designed to surface (e.g., a bug where `PRESERVATION_STRATEGY` failing incorrectly changes the reason text for a simultaneous `COLLISION_POLICY` failure would only appear in a joint state, never in either check's isolated test).

## 4. Where to spend `t=3` instead of stopping at pairwise

Per NIST's staged-strength guidance (§1), pairwise is the default, not the ceiling. The checks with a _plausible_ real coupling in docmend's own design — and therefore the first candidates for `t=3` coverage if a pairwise run does not already exercise the combination — are the three checks OQ-005's own text already treats as related:

- `PRESERVATION_STRATEGY`, `MANIFEST_WRITABLE`, and `BACKUP_DESTINATION` — because OQ-005's Agent notes explicitly discuss the case where "a manifest alone" must never substitute for a byte-preserving backend, i.e., these three checks encode a policy relationship the spec already flags as subtle, which is exactly where a 2-way sweep might miss a 3-way-only defect.

Everything else in the ~10-check list is evaluated over largely independent inputs (plan schema, tool version, source hashes, CLI flags, path resolution, encoding confidence) with no documented coupling, so pairwise coverage is the appropriate default per NIST's own cost/benefit framing rather than a shortcut.

## 5. Concrete §17.3 FR-005 test-strategy addition

Recommended replacement for the current single-line FR-005 row in §17.3, and a new §17.2 Test Strategy note:

**§17.2 addition (new row or note under "Unit / domain"):**

> Safety-gate combinatorial coverage: each of the ~10 gate checks (OQ-005) is (a) unit-tested in isolation (one check failing, all others passing), (b) covered pairwise (`t=2`) across all checks using `allpairspy`-generated parameter combinations via `pytest.mark.parametrize`, with `t=3` coverage scoped to the preservation/manifest/backup-destination trio (§4), and (c) verified for the deterministic blocking-reason contract (§3): `blocking_reason` matches the fixed priority order regardless of which checks additionally failed, and `all_failures` always contains every failing check, never just the first.

**§17.3 FR-005 row (replacing "Safety-gate refusal tests per missing strategy"):**

| Requirement ID | Test / Verification Method | Status |
| --- | --- | --- |
| FR-005 | (a) One-factor-at-a-time refusal tests, one per OQ-005 check, asserting the correct `blocking_reason` and no writes occur. (b) Pairwise (`t=2`) combinatorial sweep over all OQ-005 checks via `allpairspy`, asserting `gate_passed` iff all checks pass and `blocking_reason` always equals the fixed-priority-order first failing check regardless of which other checks also failed.
