## 1. The question in docmend's own terms

docmend already has one working review mechanism: the `plan` → review → `apply` cycle (D-006), where the plan is "the human/agent review surface" for _mechanical_ decisions (rename, encoding, newline, whitespace, skip). That mechanism works today because every mechanical decision is small, structural, and expressible without showing document body text — a plan entry says "convert `foo.txt`, detected `windows-1252` at 0.97 confidence," never a passage of prose.

WH-002 (semantic correction) and WH-005 (fuzzy dedup) break that property in two different ways:

- **WH-002** decisions are inherently about _content_: "is `their` correct here, or should it be `there`?" cannot be judged from a hash and a confidence score — a human needs to see the proposed before/after text.
- **WH-005** decisions are about _relationships between files_: "are these two noisy, drifted OCR-era texts the same document?" needs enough context (titles, sizes, maybe excerpts, similarity score) to judge, but not necessarily full document bodies.

Both need a review step that (a) works with zero live UI, (b) survives being paused and resumed across sessions like every other docmend artifact (FR-013), and (c) never silently guesses (G-005, FR-015). The research question is whether an established pattern already exists for this, short of docmend growing a TUI/GUI that would collide with §11 and NG-001.

## 2. Survey of report-driven / decision-queue review patterns

### 2.1 Two-phase report-then-act text/JSON queues (deduplication tooling)

**fclones** (Rust, actively maintained, 2,700+ GitHub stars) is the cleanest precedent for a fully headless, editable decision queue. It separates _finding_ from _acting_ into two independent commands connected only by a file: `fclones group <paths> > report.txt` writes a human-readable and machine-parseable (`--format json`/`csv`/`fdupes`) report of duplicate groups, and the operator is expected to open it in a text editor, delete or edit lines for groups they disagree with, then pipe the _edited_ file into `fclones remove`/`fclones link`/`fclones move` — the consuming command trusts the file, not a re-scan. The project's own documentation is explicit about the workflow: "the report format is detected automatically... send the report produced by `fclones group` to the standard input of `fclones remove`" [official, github.com/pkolaczk/fclones]. All destructive follow-on commands also support `--dry-run` for a second preview pass [community, x-cmd.com install guide]. This is a side-car decision queue in the most literal sense: the queue _is_ the report file, and "review" _is_ hand-editing it before the second command reads it back.

**rdfind** (C++, long-standing Debian/Ubuntu-packaged tool) uses the same shape with less machine-readability: it always writes a `results.txt` listing every duplicate found for manual review, and its `-dryrun true` flag "simulates operations without modifying the filesystem" so the results file can be inspected before any `-deleteduplicates true` / `-makehardlinks true` run is issued [official, rdfind.pauldreik.se man page; community, linuxcommandlibrary.com]. rdfind does not support re-reading an edited results file the way fclones does — its review loop is "look at results.txt, then decide what real-world command to run next," which is closer to a read-only audit report than an accept/reject queue.

### 2.2 Confidence-scored cluster export with a human-edited verdict column (fuzzy entity resolution)

The Python **`dedupe`** library (dedupeio/dedupe, MIT-licensed, actively maintained) is the closest domain match to docmend's WH-005/C.3 use case — fuzzy, probabilistic matching over noisy structured records with active-learning-trained models. Its canonical worked examples end every pipeline the same way: `deduper.partition()`/`.cluster()` returns clusters as `(record_ids, confidence_scores)` tuples, which the example code then writes back into the _original_ CSV as two new columns, `Cluster ID` and `confidence_score`, alongside all the original fields [official, dedupeio.github.io/dedupe-examples/docs/patent_example.html]. Nothing in the library auto-merges or auto-deletes; clustering only _proposes_ groupings and a numeric confidence, and every downstream action (merge, discard, keep-both) happens outside the library, driven by whatever the operator does with that CSV. **Dedupe.io** (the commercial product built on the same library) formalizes this into an explicit UI step ("a step-by-step wizard for... training, clustering and reviewing the results") [official, github.com/dedupeio/dedupe README] — but the underlying `dedupe` library itself is UI-agnostic: the CSV-with-verdict-column _is_ the decision queue, and it is trivially reviewable offline in any spreadsheet or text editor. The library's active-learning step (`dedupe.console_label`, using `y`/`n`/`u`/`f` keys) is a separate, one-time _model-training_ interaction, not the per-batch review step — it teaches the matcher's weights once, then batch clustering with confidence scores runs unattended thereafter [official/community corroborated: python4data.science tutorial and dedupeio/dedupe GitHub both describe the same `console_label` → `train` → unattended `cluster`/`partition` sequence].

### 2.3 Pessimistic quiet-mode with a segregated, resumable skip pile (batch metadata correction)

**beets** (Python music-library tagger and organizer, actively maintained) already solves a problem structurally identical to WH-002/WH-005 at 100k+-item scale: unattended, confidence-scored metadata correction over a large personal media library, where "correct" is sometimes ambiguous. Its `import` command has three review postures documented side by side: interactive (`-t`/timid, confirms even high-confidence matches), normal (prompts only on low-confidence matches), and `-q`/quiet, which "never prompt[s] for input and, instead, conservatively skip[s] any albums that need your opinion" [official, beets.readthedocs.io "Using the Auto-Tagger"]. The quiet-mode fallback is itself configurable (`quiet_fallback`, default `skip`, or `asis` to accept without correction) [official, beets.readthedocs.io/reference/cli.html]. Crucially, beets' resume/incremental-import bookkeeping treats "skipped because it needed a human" as a first-class, durable, re-visitable state distinct from "already successfully imported" — `--incremental-skip-later` explicitly controls "whether beets persist[s] skip decisions during a non-interactive import so that a user can make a decision regarding previously skipped files during a subsequent interactive import run" [official, beets.readthedocs.io/reference/cli.html]. This is the same idea as docmend's existing skip-and-report posture (FR-015) but formalized as a _two-mode_ system: unattended batch runs default to pessimistic skip, and a **separate, later, still-headless-but-interactive session** works only the skip pile — never mixing "things already handled" with "things needing a decision" in one undifferentiated report.

### 2.4 Format-identification report as a standalone audit artifact (digital preservation)

Digital-preservation tooling (DROID, Siegfried, FIDO — all used as pluggable "Format Policy Registry" identification commands inside **Archivematica**) treats the report itself as the terminal work product for a huge class of decisions, with no further "apply" step baked into the identification tool at all. A file-format identification report is explicitly designed to "serve as entry points for further activities including appraisal, identification of future work efforts" [community, Code4Lib Journal, "Fractal in detail: What information is in a file format identification report?"], and Archivematica's own architecture keeps identification (read-only, produces a report/PUID) fully decoupled from the "Format Policy" rules that later decide _what to do_ with a given identified format — an archivist edits the policy rules, not individual file decisions, and the system re-runs deterministically against the rules on each transfer [official, archivematica.org/en/docs/.../preservation-planning]. This is a weaker pattern for docmend's purposes (it has no per-item accept/reject decision file, only a report-then-configure-a-rule loop), but it corroborates the general norm in this domain: identification/detection is always a separate, read-only, report-producing phase from any state-changing action, and review happens against the report, never live inside the identification tool.

### 2.5 Patch/diff hunk-level accept-reject side files (general-purpose text correction)

The POSIX `patch` utility's `--reject`/`-r` behavior is the oldest instance of this pattern and maps almost exactly onto WH-002's "propose a text correction, let the human accept, reject, or hand-edit it" need: `git apply --reject <patch>` (and plain `patch --reject`) applies every hunk it can and writes every hunk it _cannot_ confidently apply to a sibling `<file>.rej` file in the same unified-diff format, which the operator can inspect, hand-fix, and re-apply exactly like the original patch [official, man7.org/linux/man-pages/man1/patch.1.html; corroborated independently by morphllm.com and multiple Stack Overflow patch-workflow threads]. The `.rej` file is a genuine side-car decision queue: it is plain text, uses a format every developer's tooling already understands, requires no bespoke viewer, and its mere existence _is_ the "needs review" signal — a batch run that produces zero `.rej` files needs no human attention at all. The obvious generalization for WH-002 is that docmend would not need to invent a new correction-review format: unified diff, reviewed with any pager/diff tool the operator already has, is a 50-year-old, zero-dependency, headless-native review medium.

### 2.6 Plan/apply review gate for infrastructure changes (closest architectural cousin)

Terraform's `plan` → review → `apply` cycle is architecturally the same shape docmend already committed to in D-006, so it is worth citing as the pattern's biggest real-world proof point rather than a new idea to adopt: `terraform plan` "compares your current state... with your desired state... and generates a plan of action," explicitly "to help you review and verify your configuration before applying it" and "catch any errors... before you are making unwanted changes" [official-adjacent, env0.com, corroborated by controlmonkey.io: "terraform plan is the guardrail between your code and live infrastructure... you should use `terraform plan` before every apply"]. The relevant nuance for docmend: Terraform's plan file is not hand-editable text (it is a binary/JSON blob keyed to provider schemas), so teams that need finer-grained, asynchronous review put the _human decision_ one layer up — a pull-request "approve/reject" gate in front of `apply` (the Atlantis/GitOps pattern), not inside the plan file itself. This is a useful negative data point: docmend's plan/report/manifest JSON artifacts are exactly analogous to Terraform's plan, but a hand-editable, line-level decision file (§2.1/§2.2/§2.5's pattern) is closer to what WH-002/WH-005 need than Terraform's own opaque-plan model is.

### 2.7 Append-only sidecar manifest as the review/audit trail of record (preservation packaging)

**BagIt** (RFC 8493, IETF Informational, October 2018) is the standards-track precedent for docmend's own manifest concept: a "bag" is payload files plus "tag" side-car files, at minimum one manifest file "list[ing] the payload files and checksums... for integrity verification," enabling "reliable validation of the bag's completeness and authenticity through checksum comparisons" without any GUI [official, RFC 8493 / datatracker.ietf.org/doc/rfc8493; corroborated by Wikipedia and justsolve.archiveteam.org]. BagIt has no per-item accept/reject decision step of its own — it is pure provenance/integrity bookkeeping, matching docmend's existing DR-004 manifest, not a review mechanism — but it corroborates that "plain-text, sidecar, checksum-bearing manifest files, reviewed and validated by generic tools, with no bespoke UI" is the established norm in exactly docmend's domain (large personal/institutional text collections, preservation-grade safety requirements).

## 3. Compatibility assessment with §11 / NG-001

**The core pattern — a report file that a human edits and a second command reads back — is compatible with §11 (headless CLI, no UI pages/API) and NG-001 (no reading/browsing interface) as long as three boundaries hold:**

1. **docmend never renders or paginates document content itself.** Every precedent above (fclones' report, `.rej` files, dedupe's CSV, BagIt manifests) is reviewed with tools that already exist outside the tool that produced the report — a text editor, `less`/`$PAGER`, a spreadsheet, `jq`, `diff`. docmend's job is limited to _emitting_ and _re-reading_ a structured decision artifact, exactly like it already does for DR-001–DR-004; it must not grow its own viewer, search, or paging feature, which is what NG-001's rationale ("search and reading surfaces are separate systems") actually forbids. A `docmend review corrections.json` subcommand that prints a colorized diff to stdout via `rich` (already an approved dependency, §8.6) is fine — that is reporting, the same category as the existing `--verbose` output and `report.json`; a subcommand that lets the operator scroll/browse/search across the corpus interactively would not be.
2. **The decision artifact is a new member of the existing artifact family, not a new subsystem.** All of §2.1–§2.7's precedents are essentially "one more file the pipeline reads and writes," which is exactly docmend's existing plan/report/manifest shape (DR-001–DR-004). A `corrections.json`/`dedup_review.csv` artifact consumed by a new `apply-corrections`/`apply-dedup` command (parallel to today's `plan`/`apply`) keeps the CLI surface additive and headless — no interactive prompt loop inside `apply` itself, matching the project's existing non-interactive, resumable, batch-first design (FR-013, NFR-004).
3. **A genuine, unresolved tension: WH-002 review necessarily exposes document body text in a durable artifact, which is new for this project.** Every existing docmend artifact (inventory, plan, report, manifest) is deliberately hash-and-path-only — §13.5 states this as a mitigation in force today: "logs record paths and hashes, not document content, at default verbosity." A correction-review artifact for WH-002 _cannot_ do this and still be reviewable — judging "should `teh` become `the`" requires showing the actual text. WH-005's dedup-cluster review is materially less exposed: paths, sizes, similarity scores, and maybe short excerpts are enough to judge a match without reproducing full document bodies, so it sits much closer to the existing hash/path-only posture (§2.2's `dedupe`-style CSV needs no body text beyond whatever fields are already being compared). **This asymmetry is the single most important finding of this report**: a WH-005 review-queue design is a comparatively small, low-risk extension of docmend's existing artifact model; a WH-002 review-queue design requires the owner to explicitly amend §13.4/§13.5's "artifacts never carry document content" posture (or scope the correction-review artifact's retention/handling more strictly than other artifacts) before implementation proceeds — it is not a detail to leave implicit.

## 4. Footguns and Gotchas (corroborated)

- **Human-in-the-loop review queues degrade into rubber-stamping under volume, and this is well-studied, not a hunch.** A 2024 Harvard Business School study found reviewers were "19 percentage points more likely to align with AI recommendations" when given an explanation, versus a control group with no explanation — cited independently by two separate practitioner analyses of the same finding [community/blog, tianpan.co "The HITL Rubber Stamp Problem"; blog, medium.com/@tobias_pfuetze "Why Your Approval Gate Needs Its Own AI"]. A peer-reviewed systematic review of 74 aviation/healthcare automation-bias studies (Goddard, Roudsari & Wyatt, JAMIA 2012, cited by the same secondary source) identified **workload, task complexity, and time pressure** as the primary conditions under which deference to automated recommendations intensifies [official-adjacent citation chain via medium.com/@tobias_pfuetze, tracing to the peer-reviewed JAMIA review]. For docmend, this directly threatens WH-002/WH-005: at >100k-file scale, a review queue with thousands of pending items is precisely the "high workload, low context, time pressure" condition the literature associates with rubber-stamping — the C.3 non-negotiable ("never auto-delete") is undermined in practice if the queue design makes batch-approve-everything the path of least resistance.
- **A related, independently corroborated design principle for human-approval queues at scale: exception-only patterns, not review-everything.** A practitioner discussion of AI-agent human-in-the-loop patterns converges on the same conclusion from the opposite direction: "we started with review-everything and within two weeks the human reviewer became the bottleneck and just started approving blindly to clear the queue... the approve-by-exception approach is the move" [community, Reddit r/AI*Agents, corroborated by a second independent source — workflowbuilder.io: "A workflow that runs ten thousand times a day cannot [afford per-item review] — there is no human team that can review that volume, and the gate becomes a queue that fills up faster than it drains... high-volume workflows need exception-only patterns"]. This directly supports keeping docmend's \_default* posture pessimistic-skip (§2.3, beets-style) rather than "queue everything for review": only genuinely ambiguous, above-a-confidence-floor candidates should ever reach a human decision file at all; anything the planner is confident about should already have been mechanically resolved (or skipped outright) before it reaches review.
- **A queue with no explicit timeout/abandonment policy silently accumulates stale decisions.** The same HITL-pattern literature flags this generically ("a human gate that waits indefinitely will eventually fill up with abandoned items" [community, workflowbuilder.io]) — for docmend specifically, this means a review-queue artifact needs the same staleness handling the plan artifact already has via source-hash validation (FR-003): if a file in the review queue changes on disk (or is re-planned) before its decision is acted on, the stale entry must be invalidated and re-surfaced, not silently applied against outdated content.

## 5. Design options for a future OQ (WH-002 / WH-005)

**Option A — Decision-file / verdict-column side-car queue, as a new artifact class parallel to plan/report/manifest (primary recommendation).** Extend docmend's existing plan→apply architecture with a fifth artifact pair per deferred capability: a `*-candidates` artifact (planner-generated, read-only, one row per ambiguous decision: file(s), proposed action, confidence, and — for WH-002 only — the proposed before/after text or diff) and a `*-decisions` artifact (the same rows, plus an owner-edited verdict field, initially empty/`pending`). A new `apply-corrections`/`apply-dedup` command reads only the decisions artifact and refuses to act on any row still `pending`, mirroring FR-003's stale-hash refusal and FR-005's safety-gate posture. This directly generalizes §2.1's fclones report/dedupe-CSV pattern and §2.2's `dedupe`-library verdict-column pattern into docmend's existing artifact family, keeping it headless and additive rather than inventing a new subsystem. WH-005's version of this artifact needs no document-body text (paths, sizes, similarity scores, proposed canonical) and is low-risk against §13.4/§13.5 today; WH-002's version needs body excerpts and requires the §13.4/§13.5 amendment flagged in §3 above before it can be built.

**Option B — Patch/diff hunk-level side-car queue for WH-002 specifically, reusing `.patch`/`.rej` conventions.** For semantic text corrections only, emit one unified diff per file (or one aggregate patch) via Python's stdlib `difflib`, reviewed with any diff tool/pager the operator already has (`less`, `delta`, `git apply --stat --check`), and consumed with the 50-year-old `patch --reject`/`git apply --reject` accept/partial-reject/re-apply loop from §2.5. This needs zero new file-format design work and zero new dependency, and its "needs review" signal (a non-empty `.rej` file) is self-evidently machine-checkable for resumability. It is narrower than Option A (diff review only fits text corrections, not dedup-cluster relationships) and still faces the same §13.4/§13.5 body-text exposure question, since a diff necessarily contains the corrected text.

**Option C — Segregated, resumable pessimistic-skip pile as a staging step before A/B (lowest-cost, incremental).** Before building either decision-artifact format, extend the _existing_ skip-and-report mechanism (FR-015) to distinguish "skipped: genuinely risky/binary/etc." from "skipped: needs semantic/dedup judgment" as two separate buckets in the report, mirroring beets' quiet-mode/`quiet_fallback` split (§2.3). This gives the owner triage visibility and volume data (how big is the WH-002/WH-005 backlog really?) with no new artifact schema and no body-text exposure decision yet required — it answers "how many decisions would a review queue actually need to hold" before committing to Option A or B's design, and is a natural MS-ordering predecessor to either.

**Recommendation for docmend specifically:** adopt Option C first (near-zero cost, already fits the FR-015/NFR-004 posture, informs the real scale of the problem), then Option A for WH-005 (no body-text exposure conflict, directly reuses the `dedupe`-library pattern already implicitly endorsed by C.3's "confidence scores and a review path" language), and treat WH-002's version of Option A/B as blocked on an explicit owner decision to amend or scope-carve §13.4/§13.5's "no document content in artifacts" posture — that decision should not be made implicitly by whichever option an implementer happens to pick.
