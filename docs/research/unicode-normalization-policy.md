# Unicode Normalization-Form Policy for Content and Filenames

**Date:** 2026-07-05

**Related:** GAP-45 (external tracking reference — no in-repo definition found; see [Reconciliation notes](#reconciliation-notes)) · `docs/specs/docmend.md` §2.1 (in-scope mechanical transformations), §7.1 FR-003, FR-007, FR-008, FR-009, FR-011, FR-017, §7.4 DR-001–DR-005, §8.3 D-002, §10.3 EC-006/EC-007, Appendix C.3 (WH-005 entity-resolution ladder) · `docs/open-questions.md` OQ-002 (naming policy), OQ-004 (artifact schemas), OQ-005 (safety gate), OQ-013 (frontmatter required/null/status)

**Gap it fills:** The spec normalizes encoding (to UTF-8) and line endings (to LF) as explicit mechanical transformations (FR-007, FR-008, D-002), but says nothing about _which Unicode normalization form_ the resulting text or renamed filenames should take, or whether normalization happens at all. This is not a cosmetic omission: two byte-for-byte-different UTF-8 files can represent the identical human-readable string (`é` as one precomposed code point vs. `e` + a combining acute accent), and the corpus's decades-long, multi-encoding, multi-OS history (§1, §3.1) makes this collision likely rather than theoretical. Because docmend's idempotency (FR-017), staleness detection (FR-003), and future dedup ladder (WH-005/C.3) are all hash-based, an unstated normalization policy silently determines whether those mechanisms see two representations of the same text as "the same" or "different." This report closes that gap with a concrete form, pipeline stage, and hashing recommendation.

## Bottom line and recommendation

| Question | Recommendation |
| --- | --- |
| Normalization form for content | **NFC** (Normalization Form C — canonical composition). Never NFD/NFKD for storage; never NFKC/NFKD for body text. |
| Normalization form for filenames | **NFC** on the _target_ filename produced by rename actions; leave `source.original_path` byte-exact/unnormalized for provenance. |
| Pipeline stage — content | **Transform layer**, immediately after decode-to-`str` and before newline/whitespace transforms (FR-007 → normalize → FR-008/FR-009). |
| Pipeline stage — filenames | **Planning layer**, when `target_path` is computed for a rename action (FR-002/FR-010/FR-011), not at scan/discovery time. |
| `source.hash` (DR-001/DR-002, FR-003 staleness) | Hash **raw, as-read source bytes** — pre-decode, pre-normalize. Never normalize before this hash. |
| `output.hash` (DR-003/DR-004, FR-017 idempotency) | Hash the **final written bytes** — post-decode, post-NFC-normalize, post-newline/whitespace transforms, i.e. exactly what lands on disk. |
| Dedup comparison key (WH-005/C.3 rung 2+) | A **separate, explicitly-named field**, never a substitute for `source.hash`/`output.hash`. NFC is sufficient for rung 2 (exact-content dedup) because v1 already normalizes output to NFC; reserve NFKC/casefold variants for the rung-3/4 fuzzy ladder only, with confidence scoring, never as the primary hash. |
| Scope classification | **New mechanical transform**, same tier as FR-007–FR-009 — fits within the "conservative, mechanical, form-only" boundary (§2.1, NG-003) and should be raised as a new `OQ-` (see [Reconciliation notes](#reconciliation-notes)), not silently added to code. |

## 1. What normalization actually does, and why NFC is the only defensible default here

The Unicode Standard defines two families of equivalence and four resulting normalization forms in **UAX #15** (Unicode Standard Annex #15, the normative reference): NFC and NFD express _canonical_ equivalence (the same abstract characters, differing only in whether combining marks are composed into a single code point or decomposed into base + combining sequence), while NFKC and NFKD additionally fold _compatibility_ equivalence — visually or historically related but not identical characters, such as ligatures, full-width/half-width forms, and Roman-numeral code points collapsing to their ASCII-letter equivalents. [1](http://www.unicode.org/reports/tr15) UAX #15's own worked example is directly relevant to a legacy text/HTML corpus: the ffi-ligature `ﬃ` is _not_ touched by NFC (it has no canonical mapping) but _is_ expanded to `ffi` by NFKC, and `Ⅷ` (ROMAN NUMERAL EIGHT) is left alone by NFC/NFD but rewritten to `VIII` by NFKC. [1](http://www.unicode.org/reports/tr15) Python's own `unicodedata.normalize()` documentation uses the parallel example of `Ç` (U+00C7) vs. `C`+combining-cedilla (U+0043 U+0327) to introduce exactly this canonical-equivalence problem. [2](https://docs.python.org/3/library/unicodedata.html)

For docmend, this distinction maps directly onto an existing non-goal: **NG-003 forbids editing or "improving" document meaning**, and D-002/§2.1 restrict transformations to conservative, mechanical, form-only changes. NFC/NFD normalization is exactly that — a re-encoding of the _same_ abstract character sequence, never a change to which characters are present. NFKC/NFKD are not: they can visibly alter body text (unfolding ligatures, collapsing "④" to "4", flattening full-width Japanese/Chinese punctuation to ASCII), which is a real content change, not a form change, and several of the corroborating sources below single out exactly this risk. [3](https://www.rfc-editor.org/rfc/rfc8264.html)[4](https://mbrenndoerfer.com/writing/text-normalization-unicode-nlp)

IETF's PRECIS framework (RFC 8264, the standards-track successor to Stringprep/RFC 3454, still authoritative as of this writing) makes the same call explicitly for general-purpose text handling: _"In accordance with [RFC 5198], Normalization Form C (NFC) is RECOMMENDED. Protocol designers and application developers need to understand that certain Unicode normalization forms, especially NFKC and NFKD, can result in significant loss of information."_ [3](https://www.rfc-editor.org/rfc/rfc8264.html) This is an IETF Proposed Standard, not a blog opinion, and its reasoning (canonical-only equivalence preserves meaning; compatibility folding can destroy it) is the same reasoning that applies to docmend's NG-003 boundary.

NFD is not a viable _storage_ form for docmend: it exists for internal processing (accent-insensitive search, cheap prefix/suffix stripping) and is the form macOS's HFS+ historically forced onto filenames — not a form any general text-interchange guidance recommends for stored documents. [5](https://stackoverflow.com/questions/15985888/when-to-use-unicode-normalization-forms-nfc-and-nfd) The W3C's Character Model for the World Wide Web and the HTML5 specification both converge on NFC as the required/recommended form for interchanged text content, for the same reason. [6](https://www.w3.org/TR/2004/WD-charmod-norm-20040225/)[7](https://rocketvalidator.com/html-validation/text-run-is-not-in-unicode-normalization-form-c)

**Conclusion: NFC for both content and filenames.** This aligns with D-002's philosophy (one canonical form, chosen once, applied everywhere) without crossing into the semantic-alteration territory NG-003 forbids.

## 2. Why this corpus will actually exhibit the problem (not just in theory)

Two mechanisms in docmend's own described corpus (§1, §3.1) make normalization drift a live issue rather than an edge case:

- **Multi-decade, multi-OS provenance.** The library dates to the 1990s and has been "handled and edited" across that time. Files that ever passed through a classic Mac OS or HFS+/early-APFS-era machine are prone to carrying NFD-decomposed filenames: HFS+ is a _normalizing_ filesystem that forces Form D on names, while Linux and Windows leave filenames as opaque byte/UTF-16 sequences and never normalize them. [8](https://eclecticlight.co/2021/05/08/explainer-unicode-normalization-and-apfs/)[9](https://stackoverflow.com/questions/5581857/git-and-the-umlaut-problem-on-mac-os-x) Git's own `core.precomposeunicode` setting exists specifically to paper over this seam when a repository moves between macOS and Linux/Windows — direct, corroborated evidence that this is a known, recurring interoperability failure mode for exactly the kind of file-tree-plus-version-control setup docmend's OQ-008 preservation posture may use. [9](https://stackoverflow.com/questions/5581857/git-and-the-umlaut-problem-on-mac-os-x)[10](https://www.git-tower.com/help/guides/faq-and-tips/faq/unicode-filenames/mac)
- **Legacy single-byte encodings are precomposed by construction; UTF-8 sources are not guaranteed to be.** Windows-1252 and ISO-8859-1 (both named in §1 as source encodings) have no combining-diacritic code points in their 256-slot repertoire — every accented letter they can represent (`é`, `ü`, `ñ`, …) is already a single precomposed byte, so `charset-normalizer`-decoded text from those encodings will already be NFC-equivalent for the characters those encodings can express. UTF-8 sources carry no such guarantee: a UTF-8 file produced by an old NFD-aware tool, or one that simply concatenated fragments from different origins, can legally contain decomposed sequences. This means the corpus is not uniformly one way or the other — it is exactly the mixed state normalization is designed to resolve, and it means the "encoding is already precomposed" assumption some contributors might make from the legacy-encodings framing in §1 does **not** extend to the UTF-8-sourced fraction of the library.
- **String concatenation does not preserve normalization.** Even where docmend's own transforms only append/trim ASCII whitespace, UAX #15 notes that normal forms are not closed under concatenation — joining two independently-normalized strings is not guaranteed to still be normalized, because combining-mark reordering can occur at the join point. [11](https://medium.com/@wanxiao1994/introduction-to-unicode-equivalence-and-normalization-7069eaa764d1) This is a second-order argument for normalizing late in the Transform layer (after any text is assembled) rather than trusting per-fragment normalization to compose.

## 3. Pipeline stage: content

Recommend normalizing **immediately after decode, before the newline/whitespace transforms already specified in FR-008/FR-009**, inside the pure Transform layer (§8.1 step 3, NFR-005). Concretely, the transform order becomes: decode bytes → detect/confirm encoding (FR-007) → **normalize to NFC** → normalize newlines to LF (FR-008) → whitespace transforms (FR-009) → encode to UTF-8 without BOM for the writer.

Rationale for placing it there rather than earlier or later:

- It must happen **after** decode, because normalization is defined over Unicode code points, not encoded bytes — `unicodedata.normalize()` operates on Python `str`, and doing this before decode is meaningless. [2](https://docs.python.org/3/library/unicodedata.html)
- It should happen **before** the whitespace transforms because NFC only ever affects combining-mark/precomposed-character sequences — it does not create, remove, or reclassify ASCII whitespace or line-ending characters (LF/CR/space/tab), so it cannot interact adversarially with FR-008/FR-009's blank-line collapsing or trailing-whitespace trimming. Placing it first keeps every later transform operating on a single canonical text stream, which is easier to reason about and test in isolation (NFR-005 purity). One character to flag explicitly: NFC does not touch NBSP (U+00A0) — it is not part of any canonical decomposition — so normalization has no effect on the existing whitespace-transform boundary questions already open in the spec.
- Because NFC is **idempotent** — applying it to already-NFC text is a documented no-op (`toNFC(toNFC(s)) == toNFC(s)`) — placing it unconditionally in the transform chain has zero cost on the already-precomposed legacy-encoding fraction of the corpus described in §2, and only does real work on the minority of UTF-8-sourced files that need it. [12](https://www.unicode.org/policies/stability_policy.html)

## 4. Pipeline stage: filenames

Filename normalization is a **Planning**-layer decision, not a Transform-layer one, because it only applies at the moment a `target_path` is computed for a rename action (FR-002, FR-010) — there is no "filename transform" pass independent of planning a rename. Recommend:

- Normalize the filename **stem** (not the extension token, which is ASCII already) to NFC when computing `target_path` for the `.txt`→`.md` rename.
- Leave `source.original_path` **byte-exact**, unnormalized — it is provenance, not a canonical value, and DR-001/DR-002 already require it to reflect the file as found.
- This is a mechanical, form-only change (same abstract characters, different internal representation) and therefore fits inside OQ-002's existing "mechanical vs. meaningful" naming-policy boundary alongside the extension rename it already resolves — it does not require treating this as semantic renaming (WH-001).
- **New edge case to add to §10.3**: two on-disk files whose _stems_ are canonically equivalent but byte-distinct (e.g. one NFC, one NFD) will collapse onto the same `target_path` once normalized. This must be routed through the existing FR-011 collision policy (`skip`/`fail`/`overwrite`), not treated as a new, separate mechanism — but it needs an explicit row (candidate `EC-009`) because today's §10.3 collision example (`foo.txt` vs. existing `foo.md`) does not cover a collision that is _created_ by the tool's own filename normalization rather than pre-existing on disk.
- Cross-platform caveat for OQ-008 (preservation posture): if the chosen preservation strategy is Git and the repository is ever cloned onto macOS, `core.precomposeunicode` interacts with whatever normalization docmend already applied — normalizing to NFC before the file ever reaches Git removes the ambiguity Git's setting exists to paper over, which is a point in favor of doing it in docmend rather than leaving it to whichever preservation tool is chosen later. [9](https://stackoverflow.com/questions/5581857/git-and-the-umlaut-problem-on-mac-os-x)

## 5. Idempotency and hashing: the FR-003 / FR-017 / WH-005 tradeoff

This is the crux of the research question, and the resolution follows directly from what each hash is _for_:

**`source.hash` (DR-001, DR-002; consumed by FR-003's stale-plan check) must be computed over the raw bytes exactly as read from disk — before decode, before any normalization.** FR-003's job is to detect _any_ on-disk change since the plan was made, so that apply refuses to act on stale decisions. That check is strictly cheaper and strictly more correct if it is a pure byte comparison: it requires no decode step to evaluate, it cannot itself misclassify a real edit as "no change" (which a post-normalization hash could do only in the very narrow case where an edit purely changed normalization form and nothing else — an edge case, but one a staleness gate should not paper over), and it matches how the rest of the plan/apply contract already works (DR-002 already requires "the source hashes it was decided on" to be pinned at plan time, over the same bytes planning read).

**`output.hash` (DR-003, DR-004; consumed by FR-017's idempotency check) must be computed over the final written bytes — after decode, after NFC normalization, after newline/whitespace transforms, i.e. exactly the UTF-8/LF bytes that hit disk.** FR-017 requires that re-applying the same plan, or re-planning and re-applying over already-converted output, produces zero further changes. This is only guaranteed if `output.hash` reflects the _actual_ deterministic output of the full transform chain — including normalization. Two facts make this safe rather than fragile:

1. **Determinism across runs.** NFC is idempotent by definition (§3), so re-running the full transform chain over already-converted (already-NFC) output reproduces byte-identical output, and `output.hash` matches on the second pass — exactly the FR-017 contract.
2. **Determinism across Unicode versions.** The Unicode Consortium's Normalization Stability Policy is a hard guarantee, not a convention: _"If a string contains only characters from a given version of Unicode, and it is put into a normalized form in accordance with that version, the results will be identical to the results of putting that string into a normalized form in accordance with any subsequent version of Unicode."_ [12](https://www.unicode.org/policies/stability_policy.html) This means a later Python/`unicodedata` upgrade (a newer Unicode Character Database) cannot silently change `output.hash` for text made of already-assigned characters — the one documented historical exception (Corrigendum #5, "Normalization Idempotency," affecting versions prior to Unicode 4.1) is long closed and does not apply to any currently supported Python. [12](https://www.unicode.org/policies/stability_policy.html)[1](http://www.unicode.org/reports/tr15) Recording the Unicode Character Database version alongside `docmend.conversion_version` (e.g. via `unicodedata.unidata_version`) is still worth doing for audit completeness, but it is defense-in-depth, not a required mitigation for a live instability risk.

Framed the other way: **not** normalizing before computing `output.hash` would be the actually dangerous choice for FR-017, because it would leave normalization as a _future_, not-yet-applied transform — and the day it _is_ added (which C.3 rung 1 already anticipates: "normalize text first... the v1 substrate itself materially improves match quality"), every previously-converted file whose text needed normalizing would register as a real, hash-changing mutation on the next apply. Doing it now, inside the already-idempotent mechanical substrate, is strictly cheaper than doing it later as a corpus-wide rewrite.

**Dedup/near-duplicate matching (WH-005, Appendix C.3) needs its own, separate field — never a repurposed `source.hash`/`output.hash`.** C.3's ladder already states the mechanism this report backs up with evidence: rung 1 is "normalize text first... the v1 substrate itself materially improves match quality," and rung 2 is "match exact content hashes (`source.hash`/`output.hash`) for true duplicates." Because v1's `output.hash` is now computed over NFC-normalized text (per the recommendation above), rung 2 gets normalization-invariant exact-duplicate detection **for free** — two documents whose only prior difference was normalization form will, after conversion, have identical `output.hash` values. That is a direct, positive consequence of this recommendation and requires no separate field for rung 2. Where a separate field _is_ warranted is the rung-3/4 fuzzy-matching ladder (deterministic near-duplicate rules, then confidence-scored fuzzy matching): those stages may reasonably want a coarser comparison key — NFKC plus Unicode case folding is the standard choice for search/matching use cases specifically because it deliberately erases distinctions (ligatures, width, case) that exact-duplicate detection must preserve. [4](https://mbrenndoerfer.com/writing/text-normalization-unicode-nlp) That coarser key must live in its own namespaced field (e.g. under the existing `deduplication` frontmatter object, §9), carry its own confidence/provenance per C.4's rule, and never be confused with or substituted for `source.hash`/`output.hash`.

## 6. A note on what normalization does _not_ solve: confusables and homograph risk

It is worth flagging explicitly, because it is a common and well-documented conflation: **NFC/NFKC normalization does not defend against Unicode confusable/homograph characters** (e.g. Cyrillic `а` U+0430 vs. Latin `a` U+0061), which are visually similar but canonically and compatibly _unrelated_ code points with no decomposition linking them. [13](https://hacktricks.wiki/en/pentesting-web/unicode-injection/unicode-normalization.html) That is a separate concern (Unicode's confusables data / UTS #39 skeleton algorithm), relevant to systems accepting untrusted identifiers, not to a normalization-form choice for a local, offline, single-user document-conversion pipeline. It is listed here only so that a future contributor does not read "docmend normalizes to NFC" as "docmend is safe against adversarial lookalike filenames" — it is not, and does not need to be, given NG-003/§13's threat model (path-escape and data-leakage, not adversarial content spoofing).

## 7. Verification (FR-014) implication

If content/filename NFC normalization becomes a v1 mechanical transform (see [Reconciliation notes](#reconciliation-notes)), `verify` (FR-014, OQ-006) should gain a check that converted output text and filenames are in NFC. Python's stdlib gives this cheaply and correctly: `unicodedata.is_normalized('NFC', text)` performs the UAX #15 "quick check" algorithm, which is documented to be substantially faster than normalizing and comparing, and is available in the standard library without adding a dependency. [2](https://docs.python.org/3/library/unicodedata.html)[1](http://www.unicode.org/reports/tr15)

## 8. Version and freshness notes

- Python's `unicodedata` module (checked against Python 3.14 documentation) implements `normalize(form, unistr)` over `'NFC'`, `'NFD'`, `'NFKC'`, `'NFKD'` with no API changes relevant to this decision; it is bundled with the interpreter's Unicode Character Database version and needs no separate dependency. [2](https://docs.python.org/3/library/unicodedata.html)
- UAX #15 is versioned alongside the Unicode Standard and is periodically revised for clarification, but the normalization _algorithms_ and the _stability policy_ discussed here have been fixed since Unicode 4.1 (2005) and are called out by the Consortium itself as guaranteed not to change for previously-assigned characters. [12](https://www.unicode.org/policies/stability_policy.html) No freshness risk for this recommendation.
- RFC 8264 (2017) obsoletes RFC 7564 (2015), which obsoleted the original Stringprep (RFC 3454); both successor documents agree on NFC as the recommendation, so this is a stable, multi-revision IETF consensus, not a single dated opinion. [3](https://www.rfc-editor.org/rfc/rfc8264.html)
