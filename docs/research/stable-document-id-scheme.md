# Stable Document ID Scheme Surviving Renames and Full Rewrites

**Date:** July 5, 2026

**Related:** GAP-26 (identity-recovery gap flagged for this research; not yet present as a numbered item in `docs/open-questions.md` or the spec's §21 table — see "Reconciliation notes"); OQ-002 (naming policy and identity preservation); OQ-013 (frontmatter required/null/omitted/status details); `docs/specs/docmend.md` §9 Data Model (`docmend.id` mechanical field, example shape), §7.4 DR-004 (manifest data requirement), §6 Glossary (Manifest, Preservation strategy), §8.1 Writer layer (before/after hash recording).

**Gap it fills:** The spec already names `docmend.id` as "a stable document ID surviving renames and rewrites — never depend on filename alone" (§9) and shows a placeholder value (`dmnd_0000000000000000`) in its example frontmatter, but nowhere does the spec or the open-questions backlog pin _which_ ID scheme generates that value, nor does it define what happens when a file is re-scanned and its frontmatter is missing, stripped, or corrupted — i.e., how docmend decides "this is the same document I saw before" versus "this is a new document" without being able to trust either the filename (ruled out by OQ-002) or the file content (ruled out by the "survives full rewrites" requirement, which makes content-derived/deterministic IDs unusable). This report closes that gap: it picks a concrete ID scheme and specifies the identity-recovery algorithm docmend needs on every `scan`.

---

## 1. Why content-derived IDs are disqualified up front

Before comparing schemes, one constraint eliminates a whole category of otherwise-obvious answers: **UUIDv3/v5 (namespace + name, e.g. hash of original path or content) and any ID computed as `hash(content)` cannot serve as `docmend.id`.** A full content rewrite (encoding fix, whitespace normalization, later structural conversion) changes every byte docmend would hash, so a content-derived ID silently mutates on every mutating run — the opposite of "stable." A path-derived ID fails identically under rename, which OQ-002 already rejects as a naming-policy foundation. This is why the spec is correct to require an ID that is generated once, stored, and then only ever _read back_, never _recomputed_ — the scheme choice below is really a choice among **randomly/temporally generated, opaquely stored identifiers**, and the harder engineering problem is the recovery/lookup rule (Part 3), not the generator (Part 2).

## 2. Scheme comparison

| Scheme | Bits / size | Sortability | Collision resistance | Offline generation | Stdlib/ecosystem fit for docmend (Py 3.14, zero deps) |
| --- | --- | --- | --- | --- | --- |
| **UUIDv4** | 128 bits, 36-char hex-with-hyphens string; 122 random bits | None — fully random, no time ordering | Excellent (122 random bits; RFC 9562 §5.4) | Yes, no coordination needed | `uuid.uuid4()` — already in stdlib since Python 3.x |
| **UUIDv7** | 128 bits, 36-char hex-with-hyphens; 48-bit ms Unix timestamp + version/variant bits + random tail | Lexicographic/time-ordered by construction; RFC 9562 §5.7 states v7 is "intended as monotonically ascending, creation-time-ordered, lexically sortable" [official, RFC 9562] | High — timestamp narrows the field but the random tail plus a monotonicity counter still gives strong per-millisecond uniqueness | Yes, no coordination; Python 3.14's `uuid.uuid7()` embeds a 48-bit timestamp and a **42-bit counter to guarantee monotonicity within a millisecond**, addressing the RFC's flexibility around sub-ms precision itself [official, docs.python.org] | **Native as of Python 3.14** — `uuid.uuid7()`, zero third-party dependency; exactly matches docmend's `requires-python = ">=3.14"` |
| **ULID** | 128 bits, 26-char Crockford Base32; 48-bit ms timestamp + 80 bits randomness | Lexicographic sort by design; spec states "the left-most character must be sorted first... within the same millisecond, sort order is not guaranteed" unless the monotonic variant is used [official, ulid/spec] | Very high — "1.21e+24 unique ULIDs per millisecond" per the canonical spec [official, ulid/spec]; monotonic mode increments the random tail by 1 LSB on same-millisecond collisions | Yes, no coordination | **No stdlib support** — requires a third-party package (`python-ulid`, actively maintained per PyPI/GitHub); shorter, more copy/paste-friendly string than UUID, but adds a dependency docmend's `pyproject.toml` currently has none of |
| **KSUID** | 160 bits, 27-char Base62; 32-bit second-precision timestamp + 128-bit random payload | Lexicographic sort by design (big-endian timestamp) | Highest of the four — 128 full random bits per ID, "the same collision-resistance guarantee as an entire UUID v4, even within the same second" [community, ksuid.net comparison] | Yes, no coordination | Ecosystem thin for Python: unofficial ports only (`svix/python-ksuid`, `ksuid-python`), none canonical/stdlib; 160-bit size does not fit a native UUID column if docmend ever gains a DB-backed index (it currently has none — JSON artifacts only, §9) |
| **Monotonic counter** (plain incrementing integer or per-run sequence) | Arbitrary, typically 64-bit | Perfect — strictly ordered by definition | N/A (uniqueness comes from a single authoritative counter, not entropy) | **No** — requires a durable, exclusively-owned "next value" state that must survive process crashes, be safe under `apply` resume (FR-013), and never be generated by two runs concurrently; this is the Snowflake-style "hard in distributed/offline systems" problem: a naive per-machine counter that loses its persisted state or is copied/restored from backup can hand out duplicate IDs, and reconstructing "what was the last ID issued" after a crash requires exactly the kind of coordination docmend's single-process, resumable, occasionally-interrupted batch model is designed to avoid (§7.1 FR-013, §8.1 Writer layer) | Would need a hand-rolled, file-locked counter file with its own crash-safety story — reinventing a smaller, weaker version of the manifest docmend already has |

**Bottom line on sortability/size trade-offs:** UUIDv7 and ULID are functionally near-identical — both are 128-bit, millisecond-timestamp-prefixed, lexicographically sortable identifiers; "ULID predates UUID v7 and solved the same problem in a slightly different way" and public commentary treats them as "mostly the same and have the same pros and cons" [community/blog, corroborated by two independent sources: `codetools.run` UUID guide and Reddit r/programming UUIDv7-vs-ULID discussion]. ULID's only structural edge is a shorter, no-hyphen, no-ambiguous-character string; UUIDv7's edge is that it is an IETF standard with **native Python 3.14 stdlib support**, which for a zero-dependency, offline, single-user CLI is the deciding factor, not a marginal one.

## 3. Recommendation: UUIDv7, generated with Python 3.14's `uuid.uuid7()`

**Use UUIDv7 (RFC 9562 §5.7) as `docmend.id`, generated via the Python 3.14 standard-library `uuid.uuid7()` function, stored as the canonical 36-character hyphenated string (or the `dmnd_` + hex form already sketched in the spec's example — see Reconciliation notes) in `docmend.id` frontmatter and mirrored in every manifest entry that touches the document.**

Rationale, mapped directly to docmend's stated constraints:

- **Offline, zero-dependency generation.** `pyproject.toml` currently lists `dependencies = []`; UUIDv7 is the only scheme in the comparison table requiring no third-party package at all under `requires-python = ">=3.14"`. ULID and KSUID would both add a dependency to satisfy a property (shorter string, larger entropy pool) docmend does not need — it has no database index whose B-tree page-split behavior it is trying to optimize (§9: "docmend has no database; its persistent data model is (a) the four JSON artifacts and (b) the frontmatter"). The sortability benefit that motivates UUIDv7/ULID/KSUID in database contexts is a nice-to-have here (IDs will loosely correspond to first-scan order, useful for debugging/log correlation), not a load-bearing requirement.
- **Standard, citable spec.** RFC 9562 is an IETF-published standard (May 2024) [official]; using the stdlib implementation means docmend inherits the standard's collision-resistance and monotonicity properties without maintaining any generation code itself — directly serving the project's "correctness over minimalism" and "fix root causes" posture (global instructions) and the spec's general preference for well-defined standards (CommonMark, BCP 47, JSON Schema) over bespoke formats.
- **No coordination requirement.** Unlike a monotonic counter, UUIDv7 needs no durable, exclusively-owned sequence state that must survive `apply`'s resume model (FR-013) or crash recovery (§8.1). Each ID is generated once, at first-scan/first-plan time, and never needs to be reissued or reconciled against a central authority — it is simply written into frontmatter and the manifest and read back thereafter.
- **A caveat to document, not a blocker.** RFC 9562 and independent commentary both note that v7's cross-implementation lexical sort quality "is most consistent when generated by the same library or system" [official, Wikipedia UUID article, corroborating RFC 9562 §5.7's implementation-flexibility language] and that Python's own monotonicity counter resets per-process, so ordering guarantees hold within a millisecond and within a process, not across restarts. This is irrelevant to docmend's correctness contract (`docmend.id` only needs global uniqueness plus "generally reflects first-seen order for humans skimming a manifest"), but it is worth stating explicitly in the frontmatter schema comment so a future contributor does not assume strict global ordering.

## 4. Identity-recovery algorithm on re-scan

The scheme choice only answers "how is a fresh `docmend.id` minted." The harder, previously-undocumented half of GAP-26 is: **on every `scan`, for every file docmend encounters, is this the same document as some prior run, and if so, which `docmend.id` does it get?** This must work even when frontmatter is absent (a plain `.txt` has never carried it), stripped (a user or an external tool removed it), or corrupted (unparseable YAML, duplicate keys, schema-invalid `docmend.id` field).

Recommended lookup order, evaluated per file during `scan` (read-only, per FR-001 — this step never mutates anything):

1. **Frontmatter present and valid.** Parse frontmatter per the existing strict-YAML/duplicate-key-rejection rule (C-006, DR-005). If `docmend.id` parses and validates against the frontmatter schema (well-formed UUIDv7 string, correct variant/version bits), **reuse it as-is.** This is the common case and requires no manifest lookup at all.
2. **Frontmatter missing, unparseable, or `docmend.id` fails schema validation.** Fall back to the manifest (DR-004), which already records `original_path`, `target_path`, and before/after hashes for every completed mutation, keyed indirectly by the `docmend.id` that was written into that mutation's output frontmatter at the time: a. **Path match.** Look up the manifest for any entry whose `target_path` (or, for files never yet mutated, whose `original_path`) equals the file's current path. If found and the entry's recorded `output.hash`/`source.hash` matches the current on-disk content hash, **recover that entry's `docmend.id`.** This is the cheap, common recovery path (frontmatter got stripped by an external editor, but the file was not moved). b. **Content-hash match (handles moves/renames combined with frontmatter loss).** If no manifest entry matches the current path, hash the current file body (excluding any frontmatter block, so a frontmatter-stripping edit alone does not change the match key) and search the manifest for any entry whose recorded `output.hash` (post-mutation) or `source.hash` (pre-mutation, for never-yet-mutated files) equals it. A unique match **recovers that entry's `docmend.id`** and additionally tells `plan` this is a rename to record (feeding OQ-002's path-history requirement) — the same principle Git's rename detection uses (match by content similarity across a delete/add pair, not by path) [community, corroborated by two independent sources: Stack Overflow "How does git detect similar files" and git-scm.com `gitdiffcore` documentation], simplified here to exact-hash matching since docmend does not need Git's fuzzy-similarity threshold for whole-file identity — a document that has had even its extension mechanically renamed still hashes identically pre-conversion. c. **No match (ambiguous or genuinely new).** If neither (a) nor (b) yields a unique manifest entry — including the case where multiple entries match equally well, which must never be resolved by guessing — **treat the file as a new document: mint a fresh UUIDv7 and record it.** This is a `skip-and-report`-flavored decision in spirit (FR-015's "skip and report — never guess and rewrite" doctrine extended to identity, not just content): plan/report output should flag "identity could not be recovered; a new ID was assigned" as a distinct, reviewable outcome class, not a silent success, so the owner can catch the case where a document should have matched but didn't (e.g., because it was both renamed _and_ rewritten in one external edit, which is the one combination this algorithm genuinely cannot recover — full rewrite destroys the content-hash match key, and rename destroys the path match key, simultaneously).
3. **Manifest itself missing or unreadable.** If there is no manifest at all (first-ever scan of a library) or the manifest fails its own JSON Schema validation, step 2 degenerates to "no match" for every file — every file is treated as new and gets a freshly minted ID during first `plan`/`apply`. This is expected, not an error condition, and is exactly the "cold start" case.

This gives a strict precedence order — **trust frontmatter first, trust the manifest's path/hash records second, mint new only when both are silent** — that never depends on filename alone (satisfying OQ-002's own framing) and never depends on content alone (satisfying the "survives full rewrites" requirement, because content hashing is used only as a _manifest lookup key against prior recorded hashes_, never as the ID's value).

### Worked example (synthetic, no real library data)

1. Run 1 `scan` → `plan` → `apply` converts `synthetic/notes-1987.txt` to `synthetic/notes-1987.md`, minting `docmend.id: 0198c7a4-e41a-7b6c-b39f-460873165d30` (a UUIDv7), and the manifest records `{original_path: synthetic/notes-1987.txt, target_path: synthetic/notes-1987.md, source.hash: sha256:aaa..., output.hash: sha256:bbb..., docmend.id: 0198c7a4-...}`.
2. Between runs, an external editor strips all frontmatter from `synthetic/notes-1987.md` (content otherwise unchanged) and the user also renames it to `synthetic/archive/old-notes.md`.
3. Run 2 `scan`: frontmatter is absent (step 1 fails) → path lookup fails, because `synthetic/archive/old-notes.md` matches no manifest `target_path` (step 2a fails, since the path changed) → content-hash lookup on the body (excluding the now-absent frontmatter) matches the manifest's `output.hash: sha256:bbb...` (step 2b succeeds) → `docmend.id: 0198c7a4-...` is recovered, and `plan` additionally records this as a rename (`old-notes.md` ← `notes-1987.md`) for the manifest's path-history trail.
4. If, instead, that same external edit had _also_ changed the file's body content (not just stripped frontmatter and renamed), step 2b would fail too, and the file would be assigned a brand-new UUIDv7 with an explicit "identity not recoverable, new ID assigned" flag in the report — the one unrecoverable case named above.

## 5. Existing-tool precedent for this recovery pattern

| Tool | ID mechanism | Recovery when metadata is lost | Fit for docmend's use case |
| --- | --- | --- | --- |
| **Calibre** (ebook library manager) | Random UUID stored per-book in the library database and mirrored into each book's OPF sidecar metadata (`identifiers` → `uuid`) | UUID lives in an external per-item metadata file (OPF), independent of the book's filename/path; users can bulk-search/replace the field, showing the community treats it as regenerable-if-lost rather than sacred, but the working model — durable ID stored in metadata, not derived from path or content — is exactly docmend's `docmend.id` + manifest split | Confirms the "opaque stored ID + external manifest/sidecar of record" pattern is an established one for large personal-library tools, not a docmend invention |
| **beets** (music library manager/tagger) | MusicBrainz ID (MBID), a UUID-style string, embedded in file tags plus a central SQLite database | When tags are stripped/missing, beets falls back to **acoustic fingerprinting** (AcoustID/Chromaprint) — a content-derived similarity match — to re-identify the file and re-associate the correct MBID, rather than either guessing from the filename or minting a new ID by default [community, beets FAQ + docs] | Directly analogous to Part 3's content-hash fallback: when the "obvious" identifier (tag/frontmatter) is gone, fall back to a content-derived _lookup key_ against a known-content index, not to path or to a fresh mint, before giving up |

Neither tool's precedent argues for a different generator (both predate/ignore UUIDv7 vs. ULID debates entirely), but both corroborate the two-tier "stored-ID-first, content-lookup-fallback" recovery shape recommended in Part 3.

## 6. Security and compatibility notes

- UUIDv7 embeds a creation timestamp in plaintext within the ID. For docmend this is not a privacy concern — it is a local, single-user CLI with no network exposure (§13.1: "Not applicable... it exposes no service surface and performs no network access in v1") and the timestamp is _useful_ here (it dates when docmend first touched the file, an audit-friendly property, not a liability).
- No CVEs or security advisories apply to UUIDv7/RFC 9562, ULID, or KSUID as ID _formats_ — the risk surface, if any, is in a given _implementation's_ random-number source. Python's `uuid.uuid7()` uses the same CSPRNG-backed randomness path as the rest of the `uuid`/`os.urandom` stack, so no additional hardening is needed beyond using the stdlib function as documented.
- Compatibility: `uuid.uuid7()` and UUID versions 6/8 were added to Python's `uuid` module in **3.14** specifically [official, docs.python.org — "Added in version 3.14"]; docmend's `pyproject.toml` already pins `requires-python = ">=3.14"`, so there is no floor-version conflict to manage, and no backport package is needed the way it would be for any Python version below 3.14 (the `discuss.python.org` and Reddit threads found during research both confirm no earlier CPython version ships `uuid7()`, and that the previously-circulating third-party `uuid7` PyPI package is unmaintained and pre-dates the final RFC 9562 bit layout — a reason to avoid it entirely now that stdlib support exists).

## 7. Recent changes worth tracking

- **RFC 9562** (May 2024) obsoleted RFC 4122 and formally standardized UUID versions 6, 7, and 8 — this is the primary spec citation for `docmend.id` going forward, and it explicitly documents the 16-implementation survey that motivated v7's design, plus the 48-bit-timestamp-valid-until-year-10889 headroom figure [official, rfc-editor.org].
- **Python 3.14** (2025 GA per the docs snapshot retrieved during this research) added native `uuid.uuid6()`, `uuid.uuid7()`, and `uuid.uuid8()`, plus a `--uuid`/`-u` CLI flag update to `python -m uuid` for generating versions 6–8 — this closes what had been a real gap (the community `uuid7` PyPI package was widely flagged as unmaintained and non-compliant with the final RFC bit layout) [official, docs.python.org changelog notes inline in the module reference].
- **PostgreSQL 18** added a native `uuidv7()` SQL function — not directly relevant to docmend (no database in v1, §9), but worth knowing if a future milestone (e.g., a query/index layer atop the manifest) ever adds one, since it would let a future SQL layer generate/validate the same ID format natively rather than needing an extension.

## 8. Open questions this research does not resolve

| # | Question | Why unresolved |
| --- | --- | --- |
| 1 | Should `docmend.id`'s stored string form be the raw RFC 9562 hyphenated UUID (`0198c7a4-e41a-7b6c-b39f-460873165d30`) or the `dmnd_`-prefixed compact form already sketched in the spec's example frontmatter (`dmnd_0000000000000000`)? Both encode the same 128 bits; the prefixed form is a Stripe-style typed-ID convention (cf. the `pksuid` "prefixed KSUID" pattern found during this research) that improves grep-ability and accidental-field-swap safety in logs/manifests but is not itself a standard. | Purely a presentation-layer decision on top of this report's generator choice; needs an owner call, not more research. |
| 2 | Where exactly does the manifest's content-hash lookup index live for step 2b at scale (a linear scan of a 100k-entry manifest per unmatched file could be slow)? An in-memory hash→ID index built once per `scan` invocation from the manifest is the obvious answer, but this is an implementation/performance detail properly scoped under OQ-010 (performance targets), not identity policy. | Depends on OQ-004's manifest schema being finalized first and on OQ-010's throughput targets. |
