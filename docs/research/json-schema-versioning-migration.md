# JSON Schema Versioning and Migration Policy for docmend's Artifact and Frontmatter Schemas

**Date:** 2026-07-05

**Related:** GAP-29 · [`docs/open-questions.md`](../open-questions.md) OQ-004 (artifact JSON Schemas), OQ-006 (verify semantics), OQ-009 (frontmatter emission scope), OQ-013 (frontmatter required/null/omitted/status details) · [`docs/specs/docmend.md`](../specs/docmend.md) §7.4 DR-001–DR-005, §9 Data Model, §12.1 ERR-006, §21 Open Questions and Decisions

**Gap it fills:** The spec names four artifact schemas (inventory, plan, report, manifest — DR-001–DR-004) plus a frontmatter schema (DR-005) and requires every artifact to carry a `schema_version`/`docmend.schema_version` field (§9, FR-016), but nowhere decides _what kind_ of version number that is, _what counts as a compatible change_, or _what `verify`/`apply` must do_ when the version they read does not match the version they expect — including the specific case of a schema bump landing after part of a >100k-file library has already been converted. This report closes that gap with a concrete, evidence-backed scheme sized for docmend's actual shape: an offline, single-maintainer, single-binary CLI where the "producer" and "consumer" of every schema are almost always the same installed tool version, not a distributed fleet of independent readers and writers.

## Bottom Line

Use a **two-component `MAJOR.MINOR` version string** (not full SemVer, not SchemaVer's three-part `MODEL-REVISION-ADDITION`, not a bare date) for all five schemas, versioned **independently per schema** rather than as one lock-step "data model" version. `MAJOR` means breaking (a migration function is required before older data can be treated as current); `MINOR` means "old data of this major version stays valid under the new schema" — a **backward**-only guarantee, deliberately not a forward one, because docmend's schemas are declared strict (`additionalProperties: false` per OQ-004) and strict schemas cannot honestly promise that old _code_ will accept new _data_. `$id` stays a stable, versioned-by-major pseudo-URI per schema; `$schema` stays pinned to the official Draft 2020-12 meta-schema URI and is never conflated with docmend's own `schema_version` field — these answer two unrelated questions (which JSON Schema dialect vs. which docmend contract). `verify` and `apply` refuse unconditionally on a newer-major artifact/frontmatter, refuse by default (opt-in migrate) on an older-major one, and accept-with-warning on a minor difference within the same major. Migrating an already-converted corpus is modeled as a **first-class planned action** (`frontmatter_migrate`) that flows through the existing plan → dry-run → apply → manifest → verify pipeline — never a silent side-channel rewrite — mirroring how Kubernetes' CRD `StorageVersionMigration` and Terraform's provider `StateUpgraders` both treat "upgrade already-persisted data" as an explicit, auditable operation rather than an implicit side effect of reading it.

## 1. The Versioning-Scheme Landscape

| Scheme | Shape | Compatibility signal encoded? | Real-world precedent |
| --- | --- | --- | --- |
| Date-based | `2020-12` | None — unambiguous edition marker only | The JSON Schema specification's own dialect identifiers (`draft/2020-12/schema`) [official](https://json-schema.org/understanding-json-schema/structuring); ISO standards (`ISO 8601:2004`) |
| Monotonic integer | `1`, `2`, `3` | None — a bump means "something changed," not what kind | Terraform's per-resource-type `schema_version` in `terraform show -json` output, bumped whenever a provider needs a `StateUpgrader` to run [official](https://developer.hashicorp.com/terraform/internals/json-format) |
| `MAJOR.MINOR` | `"1.0"`, `"1.1"`, `"2.0"` | Two-way: minor = additive/backward-compatible, major = breaking | Terraform's top-level `format_version` for its entire JSON output format: _"We will increment the minor version… for backward-compatible changes or additions… We will increment the major version… for changes that are not backward-compatible. Reject any input which reports an unsupported major version."_ [official](https://developer.hashicorp.com/terraform/internals/json-format) |
| Full SemVer | `1.4.2` | Three-way (major/minor/patch), designed for _code_, not data | [semver.org](https://semver.org/) — general software versioning, not schema-specific |
| SchemaVer | `MODEL-REVISION-ADDITION`, e.g. `1-0-0` | Three-way, redefined for _data compatibility_: `MODEL` = breaks all historical data, `REVISION` = may break some, `ADDITION` = compatible with all | Snowplow/Iglu's schema registry, purpose-built because "the Avro community seems to have gone down something of a rabbit hole with their schema versioning" [official/community](https://snowplow.io/blog/introducing-schemaver-for-semantic-versioning-of-schemas), [official](https://docs.snowplow.io/docs/fundamentals/schemas/versioning) |

Sourcemeta's schema-registry team, who reviewed all of the above while building schema-evolution tooling, concludes plainly: _"A single integer, incremented when a breaking change is needed, sidesteps SemVer ambiguity and gives you all the versioning room you need while keeping it simple,"_ and separately recommends **per-schema** versioning over **data-model** (lock-step) versioning: _"version only what changes, only when it changes"_ — because forcing every schema in a bundle to bump together when only one changed produces N−1 meaningless copies and noisy diffs [community/blog](https://one.sourcemeta.com/guide/evolution). Both points apply directly to docmend: its four artifact schemas plus the frontmatter schema evolve on independent timelines (a manifest bugfix has nothing to do with a frontmatter taxonomy change), and a plain integer is _almost_ enough — the one thing it can't do is let `verify`/`apply` distinguish "this is a harmless additive bump, proceed" from "this needs a migration" without an out-of-band lookup table. `MAJOR.MINOR` keeps that distinction inline in the version string itself while staying exactly as simple as Terraform's own precedent for a CLI tool emitting JSON artifacts.

## 2. The Footgun: Strict Schemas and "Additive" Changes Are Not the Same Thing

The single most important planning fact for this decision, and the reason a naïve "additive changes are always safe" policy would be wrong for docmend specifically: **JSON Schema is a constraint system, not a modeling language.** In most type systems, adding a field is free; in JSON Schema, adding a _keyword_ usually means _permitting less_, and adding a _required_ field is never additive regardless of intent. Sourcemeta's evolution guide states this directly: _"In most programming languages, adding something is backwards compatible. In JSON Schema, adding more typically means permitting less. What looks like an addition is often a restriction, and software versioning intuitions become actively misleading"_ [community/blog](https://one.sourcemeta.com/guide/evolution).

This bites hardest on **closed models** — schemas using `additionalProperties: false`, which is exactly what OQ-004's agent recommendation proposes for all four docmend artifact schemas ("keep schemas strict with `additionalProperties: false` for stable contracts"). The asymmetry is corroborated across three independent sources plus the JSON Schema project's own tracker:

- A closed schema can be **backward compatible** — old data (missing a newly-added optional field) still validates under the new schema — but it _cannot_ be **forward compatible**: new data (carrying the new field) will be rejected by any consumer still holding the old, stricter schema, because that old schema doesn't know the field exists and `additionalProperties: false` forbids it. Robert Yokota (Confluent, Schema Registry) frames this as an instance of the general robustness principle: _"For forward compatibility, one can add optional properties to an open content model or remove optional properties from a closed content model"_ — the operations are mirror images, and a closed model gets forward compatibility only by _removing_, never by adding [blog](https://yokota.blog/2025/10/07/json-schema-compatibility-and-the-robustness-principle).
- The Creek Service JSON Schema evolution series states the same rule from the opposite direction: _"Adding & removing required properties always breaks forwards and backwards compatibility for closed models. With this model it's also forward[-incompatible to add optional properties]"_ [blog](https://www.creekservice.org/articles/2024/01/08/json-schema-evolution-part-1.html).
- The JSON Schema organization's own community repository has an open GSoC 2026 project proposal to build automated tooling around exactly this gap, explicitly separating "breaking changes in schemas that allow `additionalProperties` versus those that don't" as a first-class distinction a compatibility checker must make [official](https://github.com/json-schema-org/community/issues/984).

**Implication for docmend:** keep `additionalProperties: false` — the strictness is valuable precisely because docmend cannot tolerate silently-ignored unknown fields poisoning a corpus of >100k files nobody manually reviews (this is the same C-006/FR-016 rationale that already drove duplicate-key rejection and `format`-assertion enablement). But the version-bump policy must be written to promise only what a closed schema can actually deliver: **a `MINOR` bump guarantees that data stamped with an older minor version of the same major remains valid when re-validated against the new schema** (backward compatibility of the schema toward old data). It does **not** guarantee that an older installed `docmend` binary can safely read a newer-minor artifact — and it should not try to, because for a single local CLI where one binary is always both the sole producer and sole consumer at any given moment, that forward-compatibility guarantee has no real user, while the strictness it would cost has real safety value (§8.5, NFR-004). When an old binary _does_ encounter a newer-minor artifact (e.g., after a partial tool upgrade mid-run, ERR-006's territory), refusing cleanly is not a limitation to work around — it is exactly the "skip-and-report, never guess" posture the spec already mandates for risky input (FR-015), applied to schema versions instead of file content.

## 3. `$id` Conventions

JSON Schema's own guidance is that `$id` values are **identifiers, not necessarily network locations**: _"Even though schemas are identified by URIs, those identifiers are not necessarily network-addressable... Generally, implementations don't make HTTP requests... to fetch schemas. Instead, they provide a way to load schemas into an internal schema database"_ [official](https://json-schema.org/understanding-json-schema/structuring). The value must be an absolute URI (or a relative reference resolved against a retrieval URI) without a fragment [official](https://www.learnjsonschema.com/2020-12/core/id) — but it does not have to resolve to a real page, and best practice explicitly is to always use an absolute URI to avoid retrieval-URI ambiguity [official](https://json-schema.org/understanding-json-schema/structuring).

Sourcemeta's registry-building experience converges on the same concrete recommendation this report reaches independently: put the **major version only** in a dedicated path segment (`/customers/customer/v1.json`, `/customers/customer/v2.json`), because _"a JSON Schema is uniquely identified by its URI. Two versions cannot share the same URI, thus a dedicated path component is the clearest convention"_ [community/blog](https://one.sourcemeta.com/guide/evolution). Minor bumps deliberately do **not** get a new `$id` — under this report's `MAJOR.MINOR` policy, a minor bump is backward-compatible by definition, so the same identity is correct; only a major (breaking) bump creates a genuinely new contract deserving a new URI, exactly matching how Terraform's `format_version` needs no URI at all for minor bumps and Kubernetes' CRD versions (`v1alpha1`, `v1beta1`, `v1`) are new API identities precisely because they are allowed to be non-convertible in the general case.

**Concrete recommendation for docmend**, using its real, already-public repository (`github.com/chrisdpurcell/docmend`) as the namespace rather than inventing a fictitious live domain:

```text
schemas/
  inventory.schema.json    $id: "https://raw.githubusercontent.com/chrisdpurcell/docmend/main/schemas/v1/inventory.schema.json"
  plan.schema.json         $id: "https://raw.githubusercontent.com/chrisdpurcell/docmend/main/schemas/v1/plan.schema.json"
  report.schema.json       $id: "https://raw.githubusercontent.com/chrisdpurcell/docmend/main/schemas/v1/report.schema.json"
  manifest.schema.json     $id: "https://raw.githubusercontent.com/chrisdpurcell/docmend/main/schemas/v1/manifest.schema.json"
  frontmatter.schema.json  $id: "https://raw.githubusercontent.com/chrisdpurcell/docmend/main/schemas/v1/frontmatter.schema.json"
```

- Directory `schemas/v1/` holds the current major version of every schema; a future major bump of any one schema creates `schemas/v2/<that-schema>.schema.json` with its own `$id`, while the other four stay at `v1` (per-schema, not data-model, versioning — §1 above).
- `$schema` inside every file stays `"https://json-schema.org/draft/2020-12/schema"` (the official Draft 2020-12 meta-schema URI) [official](https://json-schema.org/understanding-json-schema/structuring) — this records which JSON Schema _dialect_ the file is written in, a question completely orthogonal to docmend's own `schema_version` data field discussed next. Conflating the two (e.g., trying to encode docmend's version inside `$schema`) is a documented anti-pattern the JSON Schema org itself has debated and rejected for its own meta-schemas [official](https://github.com/orgs/json-schema-org/discussions/48).
- Because docmend never serves these files over HTTP (fully offline, §8.6/§13.1), the `raw.githubusercontent.com` URL never needs to resolve at runtime — validators load schemas from the local `schemas/` directory in-repo, exactly as `jsonschema`'s `referencing.Registry` is designed to do for exactly this "identifier without live retrieval" pattern [official/library docs](https://github.com/python-jsonschema/jsonschema/blob/main/docs/referencing.rst).

## 4. The Data-Field Distinction Docmend Already Got Right — Keep It

The spec already separates `docmend.schema_version` (which shape of data this is) from `docmend.conversion_version` (which build of the tool wrote it) — §9's example shows both fields side by side. This split is exactly what the two strongest real-world precedents researched here independently converge on:

- **Terraform** separates the top-level `format_version` (the shape of the _entire_ JSON output contract) from the per-resource `schema_version` (the shape of _one provider's_ resource data), and neither is the Terraform CLI's own release version [official](https://developer.hashicorp.com/terraform/internals/json-format).
- **Kubernetes CRDs** separate the API `apiVersion` on a resource (data shape) from the Kubernetes/controller binary version serving it — the storage version can lag or lead the serving binary, which is precisely why `conversion` webhooks and `StorageVersionMigration` exist as distinct concepts from "upgrade the cluster" [official](https://kubernetes.io/docs/tasks/extend-kubernetes/custom-resources/custom-resource-definition-versioning/), [official](https://kubernetes.io/docs/tasks/manage-kubernetes-objects/storage-version-migration/).

Do not collapse these two fields, and do not let `docmend.conversion_version` (a free-text tool version, already used for attribution per §18.3's rollback note) participate in any compatibility decision — only `schema_version` gates interpretation, exactly as §9 already states ("`docmend.schema_version` gates interpretation").

## 5. Decision Table: `verify`/`apply` Behavior on Version Mismatch

Applies uniformly to all five schemas (inventory, plan, report, manifest, frontmatter); "artifact" below covers all four run artifacts, "corpus" refers specifically to `docmend.schema_version` embedded in already-converted `.md` frontmatter.

| Scenario | `apply` behavior | `verify` behavior | Exit code family (OQ-006 taxonomy) | Precedent |
| --- | --- | --- | --- | --- |
| Artifact/frontmatter `MAJOR.MINOR` exactly matches the running tool's expected version | Proceed normally | Proceed normally | `0` | — |
| Same `MAJOR`, artifact `MINOR` _older_ than expected (e.g. tool expects `1.2`, artifact is `1.0`) | Proceed — old data is valid under the new (superset) schema by the backward-compatibility guarantee (§2) | Proceed; may annotate the report that a newer minor schema is available | `0` | Confluent's `BACKWARD` compatibility mode: new schema can read old data [official](https://docs.confluent.io/platform/current/schema-registry/fundamentals/schema-evolution.html) |
| Same `MAJOR`, artifact `MINOR` _newer_ than expected (older tool binary, newer-minor artifact) | Refuse before any mutation — this is the safety-gate case in OQ-005's checklist ("plan tool/schema version is compatible") | Refuse with a findings entry, not a silent pass | `2` (invocation/version input error) or `3` (safety refusal) per OQ-006's proposed taxonomy | Strict-schema forward-incompatibility is expected, not a bug (§2); Terraform's own policy for its `format_version` major mismatches is the same refuse-outright stance, applied here one level down at minor granularity because docmend's schemas are closed |
| Artifact/frontmatter `MAJOR` _older_ than expected | Refuse by default; require an explicit opt-in (e.g. `--migrate-schema`) that runs the registered migration chain first (§6) | Refuse (or, with `--allow-legacy`, run read-only checks scoped to what the old schema still guarantees) and report the exact version gap | `1` (findings) after migration; `3` (refusal) without it | Kubernetes: old-version CRs are served correctly but require `StorageVersionMigration` to actually rewrite them [official](https://kubernetes.io/docs/tasks/manage-kubernetes-objects/storage-version-migration/); Terraform: `StateUpgraders` run automatically only within the same provider version, otherwise the user must upgrade first |
| Artifact/frontmatter `MAJOR` _newer_ than expected (artifact from a future docmend, e.g. after a downgrade) | Refuse unconditionally — no forward migration path exists by construction | Refuse unconditionally | `3` (safety refusal) | Terraform: _"Reject any input which reports an unsupported major version"_ [official](https://developer.hashicorp.com/terraform/internals/json-format) |
| Plan file's recorded `schema_version` no longer matches what `apply` currently supports, independent of source-hash staleness (FR-003) | Refuse to start (already anticipated by ERR-006: "Plan file invalid, corrupted, or produced by an incompatible version") | N/A (verify does not consume plan files directly per FR-014) | `2` | Matches ERR-006's existing "regenerate the plan with the current tool version" recovery guidance — this report only supplies the version-comparison rule ERR-006 currently leaves unspecified |

The through-line, matching NFR-004's conservative-by-default posture: **every non-exact-match is either a safe no-op acceptance (same-major, older-minor) or an explicit, reviewable, opt-in action (everything else) — never a silent coercion.**

## 6. Migration Strategy for an Already-Converted Corpus

Two very different things need "migration," and treating them identically would be a mistake:

**Artifacts (inventory/plan/report/manifest) are transient and regenerable.** A stale `plan.json` from a retired schema major is not precious data — the source library still exists, so the ordinary recovery path is simply "re-run `scan`/`plan`" against the current schema, exactly as ERR-006 already prescribes. The only artifact with irreplaceable long-term value is the **manifest**, because it is the sole reversibility record for mutations already applied to the corpus (12.3). If the manifest schema ever takes a major bump, provide a small, tested, one-way `migrate-manifest` transform (a pure function, same testability posture as the transform layer per NFR-005) that rewrites old manifest records into the current schema _without_ touching the corpus itself — this is a low-risk, low-frequency operation because manifests are typically consulted only during a restore drill, not on every run.

**Frontmatter is embedded, scattered across >100k files, and cannot be "just regenerated."** Once OQ-009 moves frontmatter emission into scope, a `docmend.schema_version` bump inside the frontmatter schema means some fraction of a multi-GiB corpus is carrying stale frontmatter that needs an in-place rewrite — precisely the scenario Kubernetes solves with `StorageVersionMigration` (a controller that walks all persisted objects and rewrites each to the new storage version, tracked via the CRD's `status.storedVersions` field so the migration's progress and completion are observable [official](https://kubernetes.io/docs/tasks/extend-kubernetes/custom-resources/custom-resource-definition-versioning/)) and that Terraform solves with per-resource `StateUpgraders` chained hub-and-spoke through intermediate versions when a resource has drifted more than one schema version behind [official](https://developer.hashicorp.com/terraform/internals/json-format).

Docmend already has the right pipeline shape to do this safely without inventing new machinery — it just needs one new action type:

1. **Discovery** already inventories every converted file; extend it to record each file's frontmatter `docmend.schema_version` alongside the existing per-file fields (DR-001).
2. **Planning** treats a stale frontmatter version exactly like any other detected condition: emit a new planned-action kind, `frontmatter_migrate`, carrying the source hash, the old and new schema versions, and which registered migration function will run (DR-002's existing "decision provenance" requirement, C.4, already demands recording exactly this kind of input-fact/threshold/reason triple).
3. **Transform** hosts the actual migration functions as pure, individually-testable functions (old-frontmatter-dict in, new-frontmatter-dict out) — one function per `(from_major, to_major)` pair, chained if more than one major version has been skipped, exactly mirroring Terraform's chained `StateUpgraders` and Kubernetes' hub-version conversion model (convert always via a canonical "hub" version rather than every-pair-to-every-pair, to keep the function count linear instead of quadratic as majors accumulate).
4. **Writer** applies the migration through the _same_ atomic-replace, backup, and manifest path as every other mutation (D-004, FR-006) — a frontmatter migration is a content-changing write and must satisfy the same FR-005 preservation gate as any other apply. This is the single most important guardrail: a schema migration must never bypass the safety gate just because "it's only metadata."
5. **Verify** confirms every file's `docmend.schema_version` matches the currently-expected version and that the migration's manifest entries reconcile, using the same machinery FR-014 already specifies for hash/manifest consistency.

Because this reuses the existing dry-run default, staged-rollout guidance (§18.4 — filter to a small subset first, widen as confidence grows), and restore-from-manifest recovery path (12.3), a frontmatter schema migration across the full >100k-file corpus is operationally identical to any other apply run, not a bespoke one-off script that has to re-earn the tool's safety guarantees from scratch.

## 7. Concrete Recommendation Summary

1. Adopt `MAJOR.MINOR` string versioning (Terraform `format_version` precedent) for all five schemas — not integer-only (loses the compatibility signal), not full SemVer/SchemaVer (more granularity than a single-maintainer offline CLI needs), not date-based (fine for a spec dialect, useless as a compatibility signal for a project's own data).
2. Version the five schemas **independently** — a bump to `manifest.schema.json` does not touch `inventory.schema.json`'s version.
3. Keep `additionalProperties: false` (OQ-004's existing recommendation) but document the compatibility policy honestly: `MINOR` = old data stays valid under the new schema (backward only); it does **not** promise an older binary can read newer data. `MAJOR` = breaking, migration required.
4. Keep `$schema` (JSON Schema dialect, Draft 2020-12) and `schema_version` (docmend's own contract version) as two unrelated fields — never let one imply the other.
5. Put schemas at `schemas/v<major>/<name>.schema.json` with `$id` set to the corresponding `raw.githubusercontent.com/chrisdpurcell/docmend/...` URL — a real, already-public identifier requiring no live network access at validation time.
6. Wire the decision table in §5 into OQ-005's safety-gate checklist and OQ-006's `verify` exit-code taxonomy rather than treating schema-version handling as a separate concern from the safety gate.
7. Model a frontmatter schema migration as a new planned-action type (`frontmatter_migrate`) flowing through the existing plan/apply/verify/manifest pipeline — never a side-channel script — and keep artifact-schema migration cheap-and-regenerative except for the manifest, which needs its own small pure migration function for the sake of long-term restore drills.
